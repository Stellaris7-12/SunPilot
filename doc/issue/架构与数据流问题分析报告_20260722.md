# TicketAgent 架构与数据流问题分析报告

> **日期**: 2026-07-22
> **分析范围**: 数据库表结构、Agent 信息流转、Mock 工具调用、架构设计
> **核心问题**: Agent 反馈"没有合适 Tools 可供调用"

---

## 0. 业务流设想 vs 当前实现的差距（总览）

### 0.1 目标业务流（来自 `doc/planning/业务流与Agent协作.txt`）

```
发单Agent / 回单侧多Agent = 业务大脑
PageAgent = 页面执行员 + 可视化操作员
Mock Tools / MySQL = 业务系统与证据底座
人工坐席/处理员 = 最终责任人

通话记录
  → 发单Agent：提取业务字段、生成工单草稿
  → PageAgent：打开/识别发单页面，移动鼠标，自动回填字段，一键提交
  → 多Agent：分类、抽取、工具调用、风险判断、生成回单
  → PageAgent：打开工单处理页，定位证据，回填处理意见和客户回单，执行高频结案动作
  → 人工/系统门禁：高风险确认、异常接管、最终审计
```

**核心理念**：PageAgent 贯穿全流程，作为"手"来执行页面操作，多 Agent 作为"脑"来做决策。两者通过数据接口解耦：多 Agent 产出结构化指令 → PageAgent 在页面上执行。

### 0.2 当前实现的实际情况

```
通话记录 ❌ 不存在（系统只处理已创建的工单）
  → 发单Agent ❌ 完全缺失（没有从通话记录生成工单的 Agent）
  → PageAgent ❌ 完全缺失（没有任何页面自动化能力）
  → 多Agent ✅ 已实现（Classifier → Intake → Escalation → Resolution → Notification）
  → MockExecutor ✅ 代替了 PageAgent + 真实业务系统
  → 人工/系统门禁 ✅ 部分实现（状态机有 human review 状态，但确认流程不完整）
```

### 0.3 关键差距总结

| 环节 | 目标状态 | 当前状态 | 差距等级 |
|------|---------|---------|---------|
| **发单** | 发单Agent 从通话记录提取字段，PageAgent 填单 | ❌ 完全未实现 | 🔴 严重 |
| **PageAgent** | 贯穿全流程的页面执行层 | ❌ 完全未实现 | 🔴 严重 |
| **回单 Agent** | 多 Agent 协作 + 工具调用 | ✅ 基本实现，但有问题（详见下文） | 🟡 有问题 |
| **Mock Tools** | 模拟真实业务系统的接口 | ✅ 已实现 22 个 Mock 工具 | 🟢 基本可工作 |
| **人工门禁** | 高风险确认、异常接管 | 🟡 状态机已有，但确认流不完整 | 🟡 有问题 |

### 0.4 PageAgent 缺失导致的架构断层

当前 MockExecutor 直接模拟了"调用后端 API 并返回结果"的过程。但在目标架构中，MockExecutor 模拟的真实业务操作（如修改地址、补发优惠券）应该由 **PageAgent 在页面上执行**。这意味着：

1. **当前**: `ResolutionAgent 选工具 → MockExecutor.execute() → 返回 ToolResult`
2. **目标**: `ResolutionAgent 选工具 → PageAgent 打开页面 → 识别元素 → 填写/点击 → 获取结果 → 返回 ToolResult`

PageAgent 的输出格式应该与 MockExecutor 的 `ToolResult` 兼容，这样才能无缝切换。但目前 `ToolResult` 的设计完全没有考虑 PageAgent 的需求（如页面 URL、元素定位器、操作步骤序列、截图证据等）。

## 1. 数据库表与字段分析

### 1.1 当前表结构总览（13 张表）

| 表名 | 用途 | 行数评估 |
|------|------|----------|
| `tickets` | 工单主表 | 核心业务表 |
| `ai_results` | AI 处理结果（JSON 快照） | 每次处理一行 |
| `trace_steps` | Agent 执行轨迹 | 每次处理多行 |
| `tool_call_log` | 工具调用审计日志 | 每次工具调用一行 |
| `evaluations` | 评测指标 | 评测阶段使用 |
| `ticket_operation_log` | 工单操作流水 | 每次操作一行 |
| `mock_customers` | Mock 客户数据 | Mock 域 |
| `mock_cards` | Mock 卡数据 | Mock 域 |
| `mock_transactions` | Mock 交易数据 | Mock 域 |
| `mock_benefits` | Mock 权益数据 | Mock 域 |
| `mock_applications` | Mock 申请数据 | Mock 域 |
| `mock_coupons` | Mock 优惠券数据 | Mock 域 |
| `mock_permissions` | Mock 权限配置 | Mock 域 |
| `mock_tool_history` | Mock 工具调用历史（幂等） | Mock 域 |

### 1.2 发现的问题

#### 问题 1：`ai_results` 表字段冗余严重

`ai_results` 表同时存储了：
- `result_json` — 完整 JSON 快照（包含所有 Agent 输出）
- `intent_type`, `intent_label`, `intent_confidence` — 从 `result_json` 中冗余提取
- `extracted_fields_json` — 同样冗余
- `tool_name`, `tool_request_json`, `tool_response_json`, `evidence_id` — 同样冗余
- `reply_draft`, `notification_json` — 同样冗余

**风险**: 数据不一致 — 当 `result_json` 中的值与独立列不同步时，无法确定哪个是真实值。建议要么只保留 `result_json` + 少数高频查询列（如 `status`, `intent_type`），要么完全结构化。

#### 问题 2：`trace_steps` 表缺少关键字段

`trace_steps` 记录了 Agent 的执行轨迹，但缺少：
- **`input_json`** / **`output_json`** — 无法回溯 Agent 的输入输出，排查问题时只能靠日志
- **`error_message`** — 失败时没有地方存错误详情，`summary` 字段承载了太多职责
- **`llm_tokens_used`** — 无法统计 token 消耗

#### 问题 3：`tool_call_log` 表与 `mock_tool_history` 职责重叠

- `tool_call_log`: 审计视角，记录每一次工具调用
- `mock_tool_history`: 幂等视角，记录 Mock 工具的历史

两者的 `request_json` / `response_json` / `evidence_id` 字段高度重叠。在实际生产环境中，`tool_call_log` 会被保留但 `mock_tool_history` 是 Mock 专用的，应该在 Mock 模式下合二为一或明确废弃 `mock_tool_history`。

#### 问题 4：`tickets` 表字段命名不一致

- `card_last4` — snake_case
- `risk_label`, `risk_level` — snake_case
- 但工具参数和 Agent 输出中使用 camelCase（`customerId`, `couponType`）

`tickets` 表的 `card_last4` 是 `CHAR(4)` 类型，但 `mock_transactions` 表中 `card_last4` 也是 `CHAR(4)`。如果未来有虚拟卡号（如 16 位 token），这个字段会溢出。

#### 问题 5：`mock_permissions` 表未被使用

DDL 中定义了 `mock_permissions` 表，但在 `mock_executor.py` 和 `repositories.py` 中没有任何引用。说明这是一个预留但未实现的权限控制机制。

#### 问题 6：缺少 `agent_execution_log` 表

当前只有 `trace_steps` 做简要记录。实际生产需要一张专门的 Agent 执行日志表，记录：
- LLM 请求的完整 prompt（用于调试 token 超限）
- LLM 返回的原始文本（用于调试 JSON 解析失败）
- 重试次数
- Token 消耗量

## 2. Agent 信息流转分析

### 2.1 当前 Agent Pipeline 数据流

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Orchestrator.process_ticket()                 │
│                                                                      │
│  Ticket ──► ClassifierAgent ──► IntakeAgent ──► FieldEnrichment     │
│  (DB)       │                    │                │                  │
│             │ intent {type,      │ fields[]       │ 补全 customerId  │
│             │  label, conf}      │                │ couponType 等    │
│             ▼                    ▼                ▼                  │
│                                                                      │
│       EscalationAgent(1) ──► 早停检查 ──► ResolutionAgent           │
│       │                      │  needs_more_info  │                   │
│       │ checks[]             │  或 risk=medium   │ tool_name         │
│       │ can_auto_proceed     │  且未 confirmed   │ tool_params       │
│       ▼                      ▼                   ▼                   │
│                                                                      │
│       MockExecutor.execute() ──► EscalationAgent(2) ──► Notification │
│       │                          │                       Agent       │
│       │ ToolResult               │ 二次校验              │            │
│       ▼                          ▼                       reply_draft │
│                                                                      │
│       最终状态: pending_human_review / escalated / pending_info      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 Agent 间数据传递方式

每个 Agent 通过 `input_data` dict 接收上游数据，返回 dict：

| 步骤 | 输入（关键字段） | 输出（关键字段） |
|------|-----------------|-----------------|
| ClassifierAgent | `ticket_content`, `workflow_config` | `type`, `label`, `confidence` |
| IntakeAgent | `ticket_content`, `intent_type`, `intent_label`, `workflow_config` | `fields[]` |
| EscalationAgent(1) | `ticket`, `intent`, `fields[]`, `workflow_config` | `checks[]`, `can_auto_proceed`, `risk_level` |
| ResolutionAgent | `intent`, `fields[]`, **`available_tools`(Markdown)**, `workflow_config` | `tool_name`, `tool_params` |
| MockExecutor | `tool_name`, `tool_params` | `ToolResult` |
| EscalationAgent(2) | 同上 + `tool_result` | `checks[]`, `can_auto_proceed` |
| NotificationAgent | 全部上下文 + `status` | `reply_draft`, `notification` |

### 2.3 发现的信息流问题

#### 问题 7：ResolutionAgent 缺少原始工单上下文

ResolutionAgent 的输入中**没有 `ticket_content` 或 `ticket` 对象**。它只能看到 IntakeAgent 抽取后的字段。如果 IntakeAgent 漏抽或错抽了字段，ResolutionAgent 无法从原始工单中恢复。

```python
# orchestrator.py 第154-165行 — ResolutionAgent 的输入
{
    "intent": intent_result,
    "fields": extract_result.get("fields", []),  # 只有已抽取字段
    "available_tools": available_tools,           # Markdown 字符串
    "workflow_config": workflow_config,
}
# ⚠️ 缺失: ticket_content, ticket 对象
```

#### 问题 8：不存在 Agent 间回退/重试机制

Pipeline 是严格线性的，一旦某个 Agent 输出有问题，没有机制回退到上一个 Agent 重新处理。例如：
- ResolutionAgent 选错工具 → 无法让 IntakeAgent 重新提取字段
- EscalationAgent 判断需要更多信息 → 只能暂停等人工，不能让 ClassifierAgent 重新分类

#### 问题 9：EscalationAgent 每次都要重新判断"是否需要人工确认"

EscalationAgent 在 Pipeline 中被调用了**最多 2 次**（工具执行前 + 工具执行后），但两次调用的 `input_data` 格式不同（第二次多了 `tool_result`），而且逻辑中有大量重复的 if-else 分支。容易产生逻辑不一致。

#### 问题 10：Field Enrichment 只在 Pipeline 前半段执行

`_run_field_enrichment` 在 IntakeAgent 之后、EscalationAgent(1) 之前执行。但如果工具执行后发现了新的缺失字段（例如 MockExecutor 返回 "缺少某参数"），Field Enrichment 不会再次执行。

## 3. 架构与数据流格式问题

### 3.1 核心架构问题

#### 问题 11（严重）：不使用 LLM 原生 Function Calling / Tool Use

这是**最根本的架构问题**。整个系统的工具调用流程是：

```
LLM (通过 Markdown 文本认识工具) → 选出 tool_name → Python 代码执行工具
```

而不是标准的：

```
LLM (通过 tools 参数认识工具) → LLM 返回 tool_calls → Python 代码执行 → 结果回传 LLM
```

**具体表现**：

1. `base.py` 的 `call_llm()` 方法**完全没有传递 `tools` 参数**：
```python
# base.py 第67-101行
payload = {
    "model": LLM_MODEL,
    "messages": [...],
    "response_format": {"type": "json_object"},
    # ⚠️ 没有 "tools": [...] ！
}
```

2. 工具列表通过 `tool_registry.get_all_summaries()` 转成 Markdown 文本，嵌入 User Prompt 传给 LLM

3. ResolutionAgent 只是**解析 LLM 返回的 JSON 中的 `tool_name` 字段**，然后由 Orchestrator 手动调用 MockExecutor

**后果**：
- DeepSeek 模型没有被 fine-tune 过 tool-calling 格式，从 Markdown 中选工具**极不可靠**
- LLM 容易幻觉出不在注册表中的工具名
- 参数提取完全依赖 LLM 的 JSON 输出质量，没有 schema 校验
- 这就是 **"Agent 每次都说没有合适 Tools"的根本原因**

#### 问题 12：Markdown 工具列表格式不适合 LLM 解析

`tool_registry.get_all_summaries()` 生成的格式（以 22 个工具为例）：

```markdown
## 可用工具列表

### coupon.reissue - 优惠券补发
- 描述: 为客户补发指定类型的信用卡优惠券
- 分类: coupon
- 风险等级: low
- 需要确认: 否
- 参数:
  - `customerId` (string, 必填): customerId
  - `couponType` (string, 必填): couponType
  - `reason` (string, 必填): reason

### customer.update-address - 修改客户地址
...
```

**问题**：
- 22 个工具 × 人均 3-5 个参数 = 数百行 Markdown，Token 消耗大
- 参数描述都是英文 camelCase（如 `customerId`），但系统 prompt 是中文，造成语言不匹配
- I3 工具的参数 `description` 只是参数名的重复（`_param("customerId")` → description="customerId"），没有业务语义，LLM 无法理解参数含义

#### 问题 13：`tools.json` 与代码中的工具定义不同步

- `data/tools.json` 只定义了 5 个核心工具
- `definitions.py` 的 `_builtin_i3_tools()` 额外硬编码了 17 个 I3 工具
- 这导致配置文件不能反映实际的工具全集
- I3 工具的参数 `description` 全部是参数名本身（如 `customerId` 的 description 就是 "customerId"），导致 Markdown 工具列表信息量极低

### 3.2 数据格式一致性问题

#### 问题 14：camelCase 与 snake_case 混用

| 层级 | 命名风格 | 示例 |
|------|---------|------|
| 数据库列 | snake_case | `customer_id`, `card_last4` |
| Pydantic 模型 (Python) | snake_case | `customer_id`, `card_last4` |
| API 输出 (camelCase alias) | camelCase | `customerId`, `cardLast4` |
| Agent Card JSON | snake_case | `agent_id`, `input_schema` |
| tools.json 参数 | camelCase | `customerId`, `couponType` |
| workflow_config.json | camelCase | `customerId`, `newAddress` |

**后果**：
- `_normalize_tool_params()` 在 `resolution_agent.py` 中需要做特殊处理
- `enrich_params()` 在 `mock_executor.py` 中需要同时兼容两种格式
- Repository 中的 `find_customer()` 需要兼容 `customerId` 和 `customer_id`

### 3.3 架构设计问题

#### 问题 15：Orchestrator 硬编码了 Pipeline 逻辑

`orchestrator.py` 的 `process_ticket()` 方法长达 260 行，包含了所有的流程控制、状态转换、异常处理。Agent 之间的依赖关系虽然在 `agent_cards.json` 中有声明（`dependencies` 字段），但 Orchestrator 完全没有使用这些元数据来动态编排。

**后果**：
- 新增一个 Agent 需要修改 Orchestrator 核心代码
- 无法灵活配置 Agent 的执行顺序
- Agent Card 中声明的依赖关系形同虚设

#### 问题 16：没有 Agent 输出校验层

每个 Agent 的 `output_schema` 在 `agent_cards.json` 中有声明，但执行时**完全没有进行 schema 校验**。如果 LLM 返回了不符合 schema 的 JSON（如字段类型错误、缺少必填字段），下游 Agent 会直接崩溃。

## 4. Mock 工具调用问题（根因分析）

### 4.1 "没有合适 Tools 可供调用" 的根因链

```
┌─────────────────────────────────────────────────────────────────┐
│                     根因分析链路                                 │
│                                                                  │
│  ① base.py 未使用 OpenAI tools 参数                              │
│       ↓                                                          │
│  ② 工具列表以 Markdown 文本嵌入 User Prompt                       │
│       ↓                                                          │
│  ③ DeepSeek-chat 模型从 Markdown 中选工具不可靠                   │
│       ↓                                                          │
│  ④ I3 工具的 description 是参数名重复，无语义信息                  │
│       ↓                                                          │
│  ⑤ LLM 返回空 tool_name 或幻觉出不在注册表中的工具名              │
│       ↓                                                          │
│  ⑥ fallback 只覆盖 5 个核心工具（_INTENT_TOOL_MAP）               │
│       ↓                                                          │
│  ⑦ MockExecutor 收到不在注册表的 tool_name → TOOL_NOT_FOUND       │
│       ↓                                                          │
│  ⑧ 最终结果：「没有合适 Tools 可供调用」                          │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 逐层详细分析

#### 层级 1：LLM 调用缺少 `tools` 参数

```python
# ai-engine/agents/base.py 第67-101行
async def call_llm(self, system_prompt: str, user_prompt: str) -> dict:
    payload = {
        "model": LLM_MODEL,
        "messages": [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
        "response_format": {"type": "json_object"},
        "temperature": LLM_TEMPERATURE,
        "max_tokens": LLM_MAX_TOKENS,
        "timeout": LLM_TIMEOUT,
        # ⚠️ 关键缺失: "tools": [tool_definitions]
        # ⚠️ 关键缺失: "tool_choice": "auto"
    }
```

这是**根本原因**。正确的做法是将工具定义通过 `tools` 参数传递给 LLM，让 LLM 在原生 tool-calling 框架下决定调用哪个工具。当前的做法相当于让 LLM "阅读一本菜单然后口头点菜"，而不是"扫描菜单条形码"。

#### 层级 2：Markdown 工具列表的信息密度问题

当前 `get_all_summaries()` 生成的 Markdown 存在以下具体问题：

1. **描述信息缺失**：I3 工具的参数 description 是代码自动生成的：
```python
# definitions.py 第102-109行
def _param(name: str, required: bool = True, param_type: str = "string") -> dict:
    return {
        "name": name,
        "type": param_type,
        "description": name,  # ← description = name，没有业务含义！
        "required": required,
        "example": "",         # ← example 为空！
    }
```

例如 `transaction.detail-query` 的参数在 Markdown 中显示为：
```
- `customerId` (string, 必填): customerId
- `transactionDate` (string, 可选): transactionDate
- `amount` (number, 可选): amount
- `merchantName` (string, 可选): merchantName
```

LLM 看到 `description: customerId` 完全无法理解这个参数是"客户的唯一标识号"。

2. **Markdown 长度问题**：22 个工具 × 平均 3.5 个参数 = 约 200+ 行 Markdown，加上系统 Prompt 和其他上下文，总 Token 可能超过 2000（当前 `LLM_MAX_TOKENS` 的默认值），导致 LLM 可能无法完整处理。

#### 层级 3：DeepSeek-chat 模型的限制

配置文件显示：
```python
LLM_MODEL = "deepseek-chat"
LLM_MAX_TOKENS = 2000
LLM_TEMPERATURE = 0.1
LLM_TIMEOUT = 30
```

DeepSeek-chat 模型：
- 对 JSON 输出的遵循度不如 GPT-4 系列
- 在复杂上下文中选工具的准确率有限
- `max_tokens=2000` 可能不足以在返回完整 JSON 的同时还包含工具选择理由
- `timeout=30` 秒可能在复杂场景下不够

#### 层级 4：Fallback 机制覆盖不足

```python
# resolution_agent.py 第11-17行
_INTENT_TOOL_MAP = {
    "COUPON_REISSUE": "coupon.reissue",
    "CUSTOMER_ADDRESS_UPDATE": "customer.update-address",
    "TRANSACTION_DISPUTE": "transaction.query",
    "BENEFIT_QUERY": "benefit.query",
    "APPLICATION_PROGRESS_QUERY": "application.progress-query",
}
```

这个 fallback 只覆盖 5 个核心场景。17 个 I3 工具（如 `dispute.case-create`、`benefit.entitlement-query`、`card.account-status-query` 等）没有任何 fallback，**只能依赖 LLM 从 Markdown 中正确选择**。

而且 fallback 触发条件是 `not result.get("tool_name")` —— 即 LLM 必须返回空的 tool_name。如果 LLM 返回了一个**不在注册表中的幻觉工具名**（如 `"query_transaction"` 而非 `"transaction.query"`），fallback 不会触发，MockExecutor 会直接返回 TOOL_NOT_FOUND。

#### 层级 5：MockExecutor 的错误处理

```python
# mock_executor.py 第38-48行
async def execute(self, tool_name: str, params: dict) -> ToolResult:
    tool_def = self._registry.get(tool_name)
    if tool_def is None:
        return self._result(
            tool_name, False, "TOOL_NOT_FOUND",
            f"Tool {tool_name} is not registered.",
            requires_human=True, duration_ms=0,
        )
```

当工具名不在注册表中时，返回的 `failure_reason` 是 `"Tool {tool_name} is not registered."`，这条消息对前端用户来说不够友好（英文、技术性强），前端可能会转译为"没有合适 Tools"展示给用户。

### 4.3 对比：标准 Tool Calling vs 当前实现

| 维度 | 标准 Function Calling | 当前实现（Markdown + JSON） |
|------|----------------------|---------------------------|
| 工具定义格式 | JSON Schema（结构化） | Markdown 文本（非结构化） |
| LLM 如何选工具 | 模型内部 tool_choice 机制 | 解析 Markdown → 输出 JSON |
| 参数填充 | LLM 按 schema 填充 | LLM 从抽取字段中选 |
| 错误处理 | API 层面校验 schema | 无校验，运行时才发现 |
| 模型支持度 | OpenAI/DeepSeek 原生支持 | 依赖 LLM 文本理解能力 |
| 幻觉风险 | 低（受限于 tools 列表） | 高（可输出任意字符串） |
| Token 效率 | 高（工具定义不计入 output） | 低（工具列表占用 input token） |

## 5. 代码级 Bug 清单

### Bug 1：EscalationAgent 中 TRANSACTION_DISPUTE 的重复/冲突逻辑

```python
# escalation_agent.py
# 第110-119行：第一次判断（工具执行前）
if intent_type == "TRANSACTION_DISPUTE" and not tool_result:
    # ...
    return {"can_auto_proceed": True, ...}  # 允许继续

# 第217-226行：第二次判断（工具执行后，但逻辑上永远触发）
if intent_type == "TRANSACTION_DISPUTE":
    # ...
    return {"can_auto_proceed": False, ...}  # 强制升级
```

**问题**：第一个检查允许 TRANSACTION_DISPUTE 自动继续到工具执行，第二个检查又把所有 TRANSACTION_DISPUTE 升级到人工。这意味着**交易争议场景总是会先执行工具查询，然后再被升级**。工具执行的结果（真实的交易流水查询）实际上没有被用于决策——因为第217行的检查在 `tool_result` 检查之后但无视了 tool_result 的内容。

**建议**：交易争议场景中，工具执行后应根据**查询到的交易状态**来决定是否需要升级，而不是一律升级。

### Bug 2：EscalationAgent 中对 `ticket_risk == "high"` 的重复检查

```python
# escalation_agent.py
# 第81-90行：第一次检查
if ticket_risk == "high":
    return {...}

# 第206-215行：第二次检查（永远不可达）
if ticket_risk == "high":
    return {...}
```

第206行的检查永远不会被触发，因为第81行已经处理了所有 `ticket_risk == "high"` 的情况并直接 return。

### Bug 3：`enrich_params()` 的 field 填充可能覆盖正确的用户输入

```python
# mock_executor.py 第86-92行
candidate_customer_id = enriched.get("customerId") or getattr(ticket, "customer_id", "")
# ...
if customer and _is_missing(enriched.get("customerId")):
    enriched["customerId"] = customer["customer_id"]  # 覆盖
```

这里先取 `enriched.get("customerId")` 然后用 `or` 回退到 `ticket.customer_id`。如果 `enriched` 中没有 `customerId` 但 `ticket.customer_id` 有值，`candidate_customer_id` 会被正确赋值。但后面 `_is_missing(enriched.get("customerId"))` 仍然为 True（因为 enriched 中没有这个 key），所以会用查到的 customer 覆盖。这逻辑是对的，但风险在于**如果查到了多个匹配的 customer，取第一个可能有误**。

### Bug 4：`_normalize_tool_params()` 只处理 3 种参数

```python
# resolution_agent.py 第96-111行
def _normalize_tool_params(tool_params: dict) -> dict:
    for key in ("couponType", "benefitCode", "applicationNo"):
        # 正则提取
    # 只处理 verifyStatus
```

只硬编码了 `couponType`, `benefitCode`, `applicationNo`, `verifyStatus` 四种参数。17 个 I3 工具的其他参数（如 `reason`, `transactionDate`, `amount`, `merchantName`, `queryReason`, `disputeReason` 等）都没有被规范化，可能出现类型错误。

### Bug 5：`workflow_config.json` 中的 `recommended_tool` 与 I3 工具不匹配

```json
// workflow_config.json
"TRANSACTION_DISPUTE": {
    "recommended_tool": "transaction.query",  // ← 基础查询工具
    ...
}
```

但 TRANSACTION_DISPUTE 场景实际上可能需要 I3 新增的 `transaction.detail-query` 或 `dispute.case-create`。`transaction.query` 只是查询流水，而 `dispute.case-create` 才是创建争议案件。在真实场景中，交易争议应该先查流水（`transaction.detail-query`），再根据需要创建争议案件（`dispute.case-create`），而不是一步完成。

### Bug 6：`mock_executor.py` 的 `_dispatch` 方法中 `benefit.query` 和 `benefit.entitlement-query` 的处理相同

```python
# mock_executor.py 第152-157行
if tool_name in {"benefit.query", "benefit.entitlement-query", "campaign.eligibility-check"}:
    benefit = await mock_business_repository.get_benefit(...)
    return self._query_result(tool_name, "benefit", "BENEFIT_FOUND", ...)
```

三个不同的工具（权益查询/权益资格查询/活动资格校验）走完全相同的 Mock 查询逻辑，返回相同的 business_code `BENEFIT_FOUND`。这导致 Agent 无法区分这三个工具的返回结果。

### Bug 7：`enrich_params` 中 `confidence` 值固定为 0.86 或 0.55

```python
# mock_executor.py 第140-141行
"confidence": 0.86 if filled and not unresolved else 0.55,
```

这个固定值没有任何实际依据，纯粹是硬编码的。建议根据实际填充的字段比例计算。

## 6. PageAgent 集成专项分析

> 依据 `doc/planning/业务流与Agent协作.txt` 中的目标架构设想

### 6.1 PageAgent 的定义与职责

PageAgent 是"页面执行员 + 可视化操作员"，负责：
- 识别当前页面的元素（按钮、输入框、下拉菜单等）
- 找到填单/回单的位置
- 操作相应按钮、选项、以及文本框填写
- 承接前面多 Agent 处理后的结构化数据

在目标架构中，PageAgent 应该在两个环节介入：

```
环节 1 - 发单：
  发单Agent 产出 {fields, ...} → PageAgent 打开发单页面 → 回填字段 → 提交 → 工单创建

环节 2 - 回单：
  多Agent 产出 {reply_draft, notification, tool_result, ...} → PageAgent 打开处理页 → 定位证据 → 回填处理意见和客户回单 → 执行高频结案动作
```

### 6.2 当前 ToolResult 与 PageAgent 需求的不匹配

当前 `ToolResult` 的数据结构：

```python
class ToolResult(ApiModel):
    success: bool
    tool_name: str
    evidence_id: str
    action: str = ""
    business_result: str = ""
    next_step: str = ""
    requires_human: bool = False
    failure_reason: str = ""
    data: dict = Field(default_factory=dict)
    message: str = ""
    duration_ms: int = 0
```

**PageAgent 需要但 ToolResult 完全缺失的字段**：

| 缺失字段 | 用途 | 重要程度 |
|---------|------|---------|
| `page_url` | PageAgent 需要知道打开哪个页面 | 🔴 必须 |
| `page_actions[]` | 结构化的页面操作步骤（填哪个字段、点什么按钮） | 🔴 必须 |
| `field_mappings[]` | Agent 字段 → 页面表单字段的映射关系（如 `customerId` → 页面上 id 为 `#cust-id` 的 input） | 🔴 必须 |
| `screenshots[]` | 操作前后的页面截图，用于审计和人工复核 | 🟡 建议 |
| `element_selectors` | 页面元素的 CSS/XPath 选择器 | 🟡 建议 |
| `expected_page_state` | 操作后的预期页面状态，用于 PageAgent 自校验 | 🟡 建议 |

### 6.3 MockExecutor 与 PageAgent 的关系

当前的 MockExecutor 处于一个尴尬的位置：

```
当前架构：
  ResolutionAgent → MockExecutor.execute(tool_name, params) → ToolResult
   ↑ MockExecutor 直接模拟后端 API 调用

目标架构：
  ResolutionAgent → PageAgent 选 tool → PageAgent 操作页面 → 获取页面结果 → ToolResult
   ↑ PageAgent 在真实页面上操作
```

**过渡策略建议**：

1. **阶段 1（当前 → 短期）**：保持 MockExecutor 作为"后端 API 模拟"，但同时定义 PageAgent 的输入输出契约。让 MockExecutor 的输出格式兼容 PageAgent。

2. **阶段 2（中期）**：实现 PageAgent 基础能力（页面打开、元素识别、表单填写），在 Mock 环境下验证端到端流程。MockExecutor 逐步退化为 PageAgent 的 fallback。

3. **阶段 3（长期）**：PageAgent 完全接管页面操作，MockExecutor 只用于单元测试和离线评测。

### 6.4 发单 Agent 缺失分析

目标架构中的"发单Agent"负责：从通话记录提取业务字段 → 生成工单草稿。

当前系统中**完全没有发单环节**：
- 没有通话记录数据源（无对应的数据库表、API 或数据文件）
- 没有发单 Agent 的实现代码
- `POST /api/tickets` 是手动创建工单的 API，没有任何 AI 辅助
- `CreateTicketRequest` schema 要求所有字段都由调用方提供

**发单 Agent 需要的新增内容**：

| 新增项 | 说明 |
|--------|------|
| `call_records` 表或数据源 | 通话记录的存储（文本或语音转写文本） |
| `TicketCreationAgent` | 从通话记录中提取：客户号、场景、标题、优先级、风险标签等 |
| `POST /api/tickets/ai-create` | 接受通话记录文本，返回工单草稿（待人工确认） |
| PageAgent 填单指令 | 发单 Agent 产出的字段 → PageAgent 可执行的填单指令 |

### 6.5 多 Agent → PageAgent 的数据协议设计缺失

当前 5 个 Agent 的输出是松散的 dict，没有统一的数据协议来描述"接下来 PageAgent 应该做什么"。需要新增一层**指令翻译层**：

```
多Agent 产出（当前）              指令翻译层（缺失）           PageAgent 输入（缺失）
─────────────────────────────    ─────────────────    ─────────────────────────
{                                                      {
  "intent": {...},                → 翻译为 →             "page_url": "/tickets/detail?id=xxx",
  "fields": [...],                                       "actions": [
  "tool_result": {...},                                    {"type": "fill", "selector": "#reply", "value": "..."},
  "reply_draft": "...",                                    {"type": "click", "selector": "#btn-submit"},
  "notification": {...}                                    {"type": "screenshot", "label": "after_submit"}
  "status": "..."                                       ],
}                                                       "evidence_ids": ["CP20260722..."],
                                                        "verify_after": {...}
                                                      }
```

**建议**：新增一个 `PageAgentInstruction` 模型，由 Orchestrator 在 Pipeline 结束时生成，作为多 Agent 和 PageAgent 之间的标准化契约。

### 6.6 回单侧 Agent 与 PageAgent 的协作流程问题

目标流程中 PageAgent 执行完工具操作后要"定位证据，回填处理意见和客户回单，执行高频结案动作"。但当前实现中：

1. **工具执行**（MockExecutor）和**回单生成**（NotificationAgent）是分离的两个步骤
2. NotificationAgent 生成的回单（`reply_draft`）是纯文本，没有结构化为页面可执行的操作
3. 没有"证据定位"的概念 — `evidence_id` 只是一个字符串，PageAgent 不知道如何在页面上展示它
4. "高频结案动作"（如常见的填单模板、状态变更按钮）完全缺失
