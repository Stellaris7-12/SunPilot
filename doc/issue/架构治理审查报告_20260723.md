# TicketAgent 架构治理审查报告（2026-07-23）

## 结论

当前架构已从 Demo 型“多个 Agent 串行拼 dict + PageAgent ReAct 盲操作”收口为：

```text
后端 5 个业务 Agent
  -> 类型化 Agent payload / AgentCard schema 校验
  -> PipelineContext 统一编排出口
  -> AiProcessResult + PageTaskEnvelope
  -> SunPilot / SemanticPageAdapter / deterministic custom tools
  -> ReAct 仅作布局异常和歧义恢复兜底
```

这个方向可以支撑 700+ 发回单场景。关键不是为 700+ 场景分别编码页面流程，而是把业务差异留在后端 workflow/Agent，把前端收敛成页面能力层。

## 多 Agent 是否冗余

### 保留的 5 个业务 Agent

| Agent | 职责 | 是否保留 |
|---|---|---|
| `ClassifierAgent` | 场景、workflow、优先级分类 | 保留 |
| `IntakeAgent` | 字段抽取、缺失信息识别 | 保留 |
| `ResolutionAgent` | 工具选择、工具参数生成 | 保留 |
| `EscalationAgent` | 缺字段、风险、工具失败、人工门禁 | 保留 |
| `NotificationAgent` | 客户回单、内部通知、复核摘要、结案建议 | 保留 |

### 不应新增的 Agent

- 不新增 `DispatcherAgent`：当前问题不是缺一个调度 Agent，而是缺类型化上下文、统一出口和明确 PageTask 协议。
- 不新增独立 `PageAgent` 后端业务 Agent：PageAgent/SunPilot 是前端受控执行层，不做业务决策。
- 旧 `Intent/Extract/Tool/Verify/Reply` 只能作为兼容 shim，不应出现在注册表、UI 或答辩主口径。

当前代码已通过 `agent_cards.json` 暴露 5 个业务 Agent，`EscalationAgent` 内部拆分为 Guard，但没有新增第六个业务 Agent。

## Agent 数据流与字段规范

### 当前主链路

```text
Ticket
  -> TicketContext
  -> ClassifierInput
  -> IntentResult
  -> IntakeInput
  -> IntakeResult(fields)
  -> EscalationInput
  -> RiskDecision
  -> ResolutionInput
  -> ToolPlan
  -> Mock ToolResult
  -> RiskDecision(二次校验)
  -> NotificationInput
  -> NotificationBundle
  -> AiProcessResult + PageTaskEnvelope
```

### 已收口点

- 新增 `models/agent_contracts.py`，Agent 输入不再由 Orchestrator 随手拼裸 dict。
- `TicketContext` 统一生成三种上下文：文本摘要、结构化工单快照、风险/通知上下文。
- `_run_agent_step()` 已接入 AgentCard input/output schema 校验。
- `workflow_config` 已新增 `WorkflowConfig`、`WorkflowScenario`、`WorkflowField` DTO，并在加载时校验 JSON 配置；高频读取点通过统一访问器读取 scenario、required fields、recommended tool 和 notification template。
- API/前端/PageTask 使用 camelCase，数据库和 Python 内部保留 snake_case，由 Pydantic/Repository 边界转换。

### 仍需加强

- AgentCard schema 目前还是轻量 JSON Schema 子集，后续可以由 Pydantic DTO 反向生成或至少集中同步。
- `workflow_config` 运行时仍以 dict 传入 Agent 以兼容 AgentCard schema，后续可继续拆成 `ToolRoutingPolicy` 等更细粒度对象。
- 真实 DOM 行为测试仍以静态 smoke 为主，后续可引入 Playwright 覆盖实际页面填表和滚动。

## PageAgent 700+ 场景适配策略

PageAgent 不应该编码 700+ 套业务规则。正确分层如下：

| 层 | 变化频率 | 负责内容 |
|---|---:|---|
| 后端 Agent / workflow | 高 | 业务规则、场景差异、必填字段、工具选择、风险门禁 |
| PageTaskEnvelope | 中 | 页面任务协议，表达“做什么”，不表达业务决策 |
| SemanticPageAdapter | 低 | 页面语义 target，如回单区、证据区、草稿表单字段 |
| custom tools | 低 | 填表、选项、点击语义目标、定位证据、滚动区域、等待状态、停给人工 |
| ReAct | 兜底 | 目标缺失、布局不确定、控件歧义、异常恢复 |

因此 700+ 场景只需要扩展：

- workflow 配置和后端 Agent 判断；
- PageTask action/value；
- 少量页面类型的 semantic target。

不需要为 700+ 场景分别写前端流程函数。

## 前端治理结果

- AI 相关快捷动作已并入 SunPilot `AgentPanel`。
- `AppHeader` 不再暴露 AI 启动按钮。
- legacy 详情页也改为挂载 SunPilot 面板。
- `PageAssistantPanel` 保留为兼容 wrapper，不再渲染旧 AI 动作栏。
- 结构化 PageTask 优先走 deterministic executor，失败后才进入 ReAct 兜底。
- `page-agent-save-draft`、`page-agent-close-ticket` 等点击动作被确定性门禁阻断，不能绕过人工复核/结案边界。
- PageAgent deterministic executor 已接入 `page_action_logs` 操作审计，上报失败不阻断前端执行。

## 数据治理结果

- 新增 `call_records`：通话记录样本和手输 transcript 先入库，再用于发单草稿生成。
- 新增 `ticket_drafts`：记录发单草稿、PageTask、pageTaskHints、缺失字段、关键字段和置信度。
- 新增 `page_action_logs`：记录 SunPilot/PageAgent 每步确定性页面动作、target、输入输出、耗时和 stopReason。
- 新增 `agent_execution_log`：记录每个后端业务 Agent 的 input/output、状态、耗时和错误信息。
- 扩展 `trace_steps`：新增 `input_json`、`output_json`、`error_message`、`duration_ms`，支持后续回放和排障。

## 按优先级修改清单

### P0：必须保持

1. 后端业务 Agent 只保留 5 个，旧 shim 不进注册表。
2. PageAgent 不做业务决策，不直接保存、结案、转派或覆盖主系统状态。
3. PageTask `requiresHumanBeforeSubmit=true` 时禁止点击提交类目标。
4. Agent 输入输出必须走 DTO + AgentCard schema 校验。

### P1：下一阶段建议

1. 将 AgentCard schema 与 Pydantic DTO 做集中同步，降低双写漂移。
2. 将 `workflow_config` 继续细分为 `ToolRoutingPolicy` 等对象。
3. 为 PageTask deterministic executor 增加真实 DOM 行为测试。

### P2：演示与答辩增强

1. 补架构图和数据流图。
2. 准备低风险、中风险、高风险/工具失败三条验收链路。
3. 将“700+ 场景适配”口径固化为：业务规则扩 workflow，页面执行扩 semantic targets，不扩 Agent 数量。

## 当前剩余风险

当前主干治理已完成并通过 smoke。剩余风险主要是静态 smoke 不能替代真实浏览器 DOM 操作测试，以及 AgentCard schema 与 DTO 仍存在双写同步成本。
