# TicketAgent Demo 讲解稿

## 一、我们要解决什么问题

信用卡工单处理的痛点，不是“有没有系统”，而是很多环节仍然依赖人工判断和人工搬运信息。

客户提出问题后，坐席或后台人员通常需要：

- 阅读客户描述或通话记录。
- 判断属于哪类工单。
- 补齐客户号、卡号、交易金额、地址等业务字段。
- 登录或查询多个业务系统。
- 判断是否可以直接处理，还是必须转人工复核。
- 最后再组织一段规范、准确、可追溯的客户回复。

这会带来三个直接问题：

| 问题 | 影响 |
|------|------|
| 处理慢 | 客户等待时间长，工单积压明显 |
| 人工重复多 | 坐席和后台人员花大量时间做复制、查询、填写 |
| 回复不稳定 | 处理结果和回单质量依赖个人经验 |

所以本 Demo 的核心不是单纯“识别客户说了什么”，而是展示：工单进入系统后，AI 能不能继续往后走，把处理、回单和风险兜底一起串起来。

## 二、项目目标

TicketAgent 的目标是做一个面向信用卡工单的智能处理工作流。

一句话概括：

```text
让低风险工单自动处理，让中风险工单人机协作，让高风险工单自动拦截升级。
```

我们希望达到的效果：

- 客户诉求能被快速识别。
- 工单关键信息能自动提取。
- 常规低风险事项能自动处理。
- 敏感事项先让人工确认，再继续处理。
- 高风险事项自动转人工，避免误操作。
- 每一步处理都有过程记录和证据编号。
- 最后能用指标说明系统节省了多少时间、处理是否准确。

## 三、整体业务流程

完整流程如下：

```text
客户诉求 / 通话文本 / 工单原文
  ↓
识别工单类型
  ↓
提取关键业务信息
  ↓
判断风险等级
  ↓
调用业务能力处理
  ↓
生成客户回单
  ↓
自动结单或人工复核
  ↓
沉淀处理记录和评测指标
```

系统不是一味追求“全自动”，而是根据风险分级处理：

| 风险等级 | 处理方式 | 示例 |
|----------|----------|------|
| 低风险 | 自动处理，自动生成回单 | 优惠券补发、进度查询 |
| 中风险 | 人工确认后继续处理 | 地址修改、资料变更 |
| 高风险 | 自动拦截并升级人工 | 交易争议、疑似盗刷 |

这种方式更适合金融业务：该自动的自动，该确认的确认，该拦截的拦截。

## 四、Demo 展示什么

### 场景一：低风险自动处理

客户反馈活动达标后优惠券未到账。

系统会自动完成：

1. 判断这是“优惠券补发”类工单。
2. 提取券类型、客户信息、补发原因。
3. 判断风险较低，可以自动处理。
4. 调用补券工具，返回处理成功和证据编号。
5. 自动生成客户回复。

展示价值：

- 原本需要人工查询和补发，现在可以自动完成。
- 回单中带有证据编号，方便追溯。
- 低风险工单可以从“排队等待”变成“即时处理”。

### 场景二：中风险人机协作

客户要求修改账单寄送地址。

系统会自动完成前半段判断，但不会直接执行敏感操作：

1. 判断这是“资料修改”类工单。
2. 提取新地址和身份核验状态。
3. 识别为中风险操作。
4. 暂停流程，请人工确认。
5. 人工确认后，系统继续完成地址修改和回单。

展示价值：

- AI 负责整理、判断和准备操作。
- 人只在关键风险点确认。
- 敏感业务不盲目自动执行，符合合规要求。

### 场景三：高风险自动拦截

客户反馈某笔交易不是本人消费。

系统会识别为交易争议或疑似盗刷风险，直接升级人工处理。

展示价值：

- 高风险事项不自动处理。
- 系统可以主动识别风险并兜底。
- 自动化不是替代风控，而是帮助风控更早介入。

## 五、产品模块说明

### 1. 工单工作台

用于查看待处理工单、风险等级、处理状态和客户诉求。它是业务人员进入系统的主界面。

核心价值：快速定位工单，减少人工翻找。

### 2. 工单详情

展示客户信息、工单内容、场景标签和当前状态。

核心价值：让处理人员一眼看到上下文。

### 3. AI 处理过程

实时展示系统正在做什么，例如：

```text
识别客户诉求
提取业务要素
完成风险校验
调用业务能力
生成客户回复
```

核心价值：让业务人员看得懂 AI 的处理过程，而不是只看到一个黑盒结果。

### 4. 处理结果与证据

展示系统识别出的意图、抽取出的字段、风险判断、工具处理结果和证据编号。

核心价值：处理过程可解释、可追溯。

### 5. 回单生成

系统根据处理结果生成标准化客户回复。

核心价值：减少人工组织话术时间，提高回复一致性。

### 6. 人工确认

用于中风险操作，例如修改客户资料、涉及金额或身份核验的业务。

核心价值：把人工从全流程操作变成关键节点审核。

### 7. 效果看板

展示处理准确率、字段完整率、工具命中率、风险拦截率和处理耗时。

核心价值：让系统效果可量化，而不是只靠演示观感。

## 六、效果如何衡量

本项目建议从两个方向衡量效果：准确性和效率。

### 准确性指标

| 指标 | 说明 |
|------|------|
| 意图识别准确率 | 是否判断对客户诉求类型 |
| 字段完整率 | 是否提取到处理所需字段 |
| 工具命中率 | 是否选择了正确处理动作 |
| 风险拦截率 | 高风险工单是否被正确升级 |
| 回单可用率 | 生成回复是否可直接使用或少量修改 |

### 效率指标

| 指标 | 说明 |
|------|------|
| 平均处理耗时 | AI 完成一次处理链路的时间 |
| 人工介入次数 | 每张工单需要人工参与几次 |
| 自动处理率 | 低风险工单中可自动完成的比例 |
| 节省时间 | 相比传统人工处理节省的时间 |

对外表达时，可以用一句话说明价值：

```text
把低风险工单从人工排队处理，变成 24 小时可自动处理；把人工精力集中到中高风险和复杂问题上。
```

## 七、业务价值

### 对客户

- 更快得到处理结果。
- 减少重复沟通。
- 常规问题不必长时间等待。

### 对坐席和后台人员

- 减少复制、查询、填写、组织话术等重复劳动。
- 只在风险确认、异常处理、复杂判断时介入。
- 工作重点从“操作”转向“审核和决策”。

### 对管理人员

- 工单处理过程可视化。
- 处理结果可追溯。
- 自动化效果可量化。
- 有利于后续推广到更多业务场景。

## 八、为什么这个方案适合当前阶段

当前阶段最重要的是跑通业务闭环，而不是一开始就追求复杂平台化。

所以本 Demo 的取舍是：

- 先做文本工单闭环，语音作为后续增强入口。
- 先用模拟工具展示跨系统处理能力，后续再接真实业务系统。
- 先把低、中、高风险三条路径讲清楚，再扩展更多场景。
- 先让业务效果可见，再逐步增强技术架构。

这样可以确保 Demo 稳定、重点清楚，也更容易让客户和领导理解价值。

## 九、后续扩展方向

后续可以逐步增强：

| 方向 | 说明 |
|------|------|
| 语音入口 | 客户通话转文字后直接生成工单并处理 |
| 更多场景 | 扩展到申请进度、协商还款、挂失、额度调整等 |
| 真实系统对接 | 将模拟工具替换为真实业务接口 |
| 自动评测 | 持续评估准确率、耗时和人工节省 |
| 内嵌工作流 | 嵌入现有工单系统，作为侧边栏或插件使用 |
| 跨系统 Agent | 未来支持更多系统间协同处理 |

## 十、结尾总结

TicketAgent 展示的是一个从“人工处理工单”走向“AI 协同处理工单”的业务闭环。

它的核心价值不是替代所有人，而是：

```text
让 AI 处理标准化、重复性、低风险工作；让人专注在确认、判断和复杂问题上。
```

最终希望实现：

- 低风险自动处理。
- 中风险人机协作。
- 高风险自动升级。
- 全流程可解释、可追溯、可评测。

这就是本 Demo 想呈现的业务价值。
