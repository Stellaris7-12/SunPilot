# 任务计划：TicketAgent 核心闭环开发路线

## 目标

把当前 TicketAgent 从“可运行 Demo”推进到“可演示、可测评、可解释的信用卡工单智能处理闭环”。

本阶段不追求一次性做完整企业级工单系统，也不把 A2A、PageAgent、真实语音识别做成主线。主线是先让系统稳定完成：接收工单/通话文本、识别诉求、补齐信息、分类定级、调用业务能力、生成回单、通知/回访、必要时升级人工，并能用数据集和指标证明效果。

## 核心原则

1. 先跑通业务闭环，再增加进阶能力。
2. 先稳定后端契约和 Agent 输出，再优化前端呈现。
3. 先用 Mock Tool/API 打通执行链路，再接 MCP、PageAgent 或真实系统。
4. 先做中等规模、高质量标注数据和评测脚本；样本优先服务场景覆盖、回归验证和指标展示，不把数量本身作为目标。
5. 先沿用当前手写 Orchestrator，等分支和状态管理复杂后再评估 LangGraph。
6. 风险分级不作为独立核心模块，只作为分类、执行、升级策略中的业务规则。

## 当前代码与新 Agent 重构口径

| 新业务 Agent | 职责 | 代码重构目标 | 当前策略 |
|-------------|------|--------------|----------|
| Intake Agent / 接单与信息提取 | 接收工单、通话文本、表单输入，抽取标题、描述、用户、关键字段；信息不足时生成追问 | `ExtractAgent` 重构为 `IntakeAgent` | P0 直接采用业务命名，旧文件短期做兼容 shim |
| Classifier Agent / 分类与优先级判定 | 判断工单类型、业务场景、处理优先级；规则先行，LLM 处理复杂表达 | `IntentAgent` 重构为 `ClassifierAgent` | P0 核心 |
| Dispatcher Agent / 智能派单 | 根据类型、紧急程度、技能、负载，把工单派给人、团队或 Agent | 当前暂无完整实现 | P2 进阶，不抢主线 |
| Resolution Agent / 解决方案与执行 | 查询知识库，调用接口/API/Mock Tool，未来接 MCP 或 PageAgent 执行业务动作 | `ToolCallingAgent` 重构为 `ResolutionAgent`，继续使用 `MockExecutor` + `ToolRegistry` | P0 核心，必须有审计证据 |
| Notification Agent / 通知与回访 | 生成处理说明、客户回单、状态通知、满意度回访 | `ReplyAgent` 重构为 `NotificationAgent` | P0/P1，先做回单和状态通知 |
| Escalation Agent / 升级与兜底 | 处理字段缺失、工具失败、SLA 即将超时、合规或异常场景，转人工或升级 | `VerifyAgent` 重构为 `EscalationAgent`，继续结合状态机和人工确认逻辑 | P0 先做兜底策略，不单独炫技 |

## 推荐主流程

```text
多渠道输入/通话文本
  -> Intake Agent：接单、字段抽取、缺失信息提示
  -> Classifier Agent：工单类型、业务场景、优先级判定
  -> Resolution Agent：选择处理方案，调用 API/Mock Tool，记录证据
  -> Notification Agent：生成回单、状态通知、回访提示
  -> Escalation Agent：贯穿全流程，处理异常、超时、失败、人工升级
  -> 工单结案/人工复核/转派
```

Dispatcher Agent 不放入当前核心闭环。需要体现“派单中心”时，先用规则或占位说明表达，等核心闭环稳定后再实现工程师/团队/Agent 的动态匹配。

## 模块 H：Page Agent 业务化改造（P1 优先）

目标：把当前已落地的前端页面助手升级为“信用卡人工发单后处理场景下的受控页面执行层”。信用卡转人工业务主要来自人工客服电话后的二线发单；App、智能客服或语音输入能直接解决的诉求不进入当前核心闭环。因此模块 H 不做语音输入模块，不做通用网页机器人，而是聚焦人工发单后的后续处理：让 Page Agent 消费工单业务上下文、执行白名单页面动作、辅助填单/回单/定位证据、留下动作审计，并且严格服从现有状态机、人工确认和结案接口。

核心判断：
- 当前 Page Agent 已部分实现，主要形态是 `frontend/src/components/ai/PageAssistantPanel.vue`，并已挂载在 `frontend/src/views/TicketDetailView.vue`。
- 它目前没有纳入后端 Orchestrator 主编排；后端主链路仍是 `ClassifierAgent -> IntakeAgent -> EscalationAgent -> ResolutionAgent -> NotificationAgent`。
- Page Agent 不应被做成第六个后端业务 Agent，也不应在当前阶段变成通用网页机器人。
- 阿里 `page-agent` 能力强，但 TicketAgent 当前不直接二开/嵌入完整工程；采用 A+B 路线：先手搓轻量 Page Assistant，再借鉴其 `PageContext`、动作 DSL、白名单工具、观察-计划-执行-验证循环和动作日志思路。
- 通用 Page Agent 通常不理解信用卡工单业务，不天然具备风险审计、权限门禁、状态机和证据编号约束；TicketAgent 的差异化不在“更会点网页”，而在信用卡工单闭环、业务专业性、状态机、工具审计、证据编号、人工确认、回单通知和可评测指标。
- Page Agent 的安全与权限是产品优势点：任何写入、发送、结案、资料修改、拒付、通知客户等动作都必须通过白名单、风险分级和人工确认；不得出现“更新草稿时误点发送”这类未授权行为。
- 如果外部页面点击后进入子页面找不到目标、选错元素、多次操作失败或超时，Page Agent 必须停止并转人工告警，而不是继续猜测点击。
- 产品形态要尽量贴近真实企业生产场景：Page Agent 不是独立炫技模块，而是嵌入完整企业工单处理系统中的右侧受控辅助层。

### H1：Page Agent 业务化 MVP（当前第一优先级）

目标：把 Page Agent 从“按钮式页面助手”升级为“受控页面动作编排层”。它只操作当前工单详情页内的低风险页面动作，不直接改变权威业务状态，不绕过 `/confirm-action` 和 `/close`。

开发方案：
- [ ] 梳理企业工单壳的页面区域、锚点和当前可执行动作：工单详情、客户/卡片/交易/权益信息、回单工作区、证据卡片、人工确认、结案复核、处理日志。
- [ ] 定义 `PageContext`：`ticket`、`aiResult`、`notification`、`traceSteps`、`toolCalls`、`missingFields`、`riskFlags`、`closureSuggestion`、`currentDraft`、`uiAnchors`、`allowedActions`、`disabledReasons`。
- [ ] 定义受控 `PageAction` DSL：`fill_reply_slot`、`apply_reply_template`、`scroll_to`、`focus_panel`、`locate_evidence`、`locate_missing_fields`、`open_business_log`、`open_tool_audit`、`prepare_human_confirm`、`prepare_close_review`、`start_ai_process`。
- [ ] 新增或抽象 `PageActionRunner`，把每个 `PageAction` 映射为明确的前端函数、现有 API 或页面滚动/聚焦动作；不开放任意 DOM index 点击，不启用任意 JavaScript 执行。
- [ ] 将 `PageAssistantPanel.vue` 从静态按钮面板升级为动作入口：根据 `PageContext.allowedActions` 展示可执行动作，根据 `disabledReasons` 展示不可执行原因。
- [ ] 增加 `PageActionLog`：记录 `action`、`input`、`targetPanel`、`result`、`duration`、`retryCount`、`operatorConfirmRequired`、`relatedEvidenceId`、`createdAt`，并与后端 Agent Trace 区分展示。
- [ ] 对敏感动作做权限门禁：人工确认必须走 `/api/tickets/{ticket_id}/confirm-action`，实际结案必须走 `/api/tickets/{ticket_id}/close`，Page Agent 只能准备页面和填充草稿。
- [ ] 页面状态只来自受控业务上下文，不把完整 DOM、客户敏感信息或任意页面快照直接发送给 LLM。
- [ ] 增加 Page Agent Policy：极高风险动作完全禁止；中风险动作必须人工确认；低风险动作只允许白名单页面内的填充、定位、展开、聚焦和草稿生成。
- [ ] 前端补充验证场景：自动成功待复核、待补充、待人工确认、高风险升级、工具失败、UNKNOWN 升级。

验收标准：
- 坐席能在工单详情页一键填入 AI 回单、定位证据、定位缺失字段、进入人工确认或结案复核区域。
- Page Agent 的动作由 `PageContext` 和 `allowedActions` 控制，不直接改业务状态。
- Page Agent 不接管语音入口，也不处理 App/智能客服自助已解决场景；它只服务人工发单后的二线处理。
- 页面助手动作日志与后端 Agent Trace 边界清晰，能展示“坐席页面辅助”但不污染工具审计证据链。
- 模块 H 汇报时能讲清：TicketAgent Page Agent 是信用卡业务受控执行层，不是通用网页自动化工具；相对通用 Page Agent 的优势是业务理解、权限控制、风险审计、证据链和人工确认。
- 修改前端后运行 `cd frontend && npm run build` 通过；若只改文档，则检查 Markdown 标题层级和模块顺序正确。

### H2：Page Agent 动态表单与半自动填充（H1 稳定后）

目标：在 H1 的受控动作框架上，扩展到更有业务价值的半自动页面填写。所有写入动作采用“预览 -> 人工确认 -> 写入页面”，不直接提交业务结果。

开发方案：
- [ ] 定义可填充字段清单：回单草稿、客户追问话术、人工确认摘要、内部通知摘要、资料变更复核字段、争议交易说明字段。
- [ ] 为每类字段定义来源优先级：确定性后端结果优先，LLM 文案只作为草稿，不覆盖证据编号、失败原因、状态和结案规则。
- [ ] 增加填充预览面板，展示字段来源、置信度、缺失原因和人工确认入口。
- [ ] 支持一键写入页面但不自动提交；提交、确认、结案继续走现有业务按钮和 API。
- [ ] 引入轻量 observe-plan-execute-verify 循环：观察 `PageContext`，生成动作计划，执行白名单动作，验证页面结果并写入 `PageActionLog`。
- [ ] 支持自然语言页面指令的窄域解析，例如“帮我填回单”“定位证据”“看一下为什么不能结案”；解析失败时退回明确按钮。
- [ ] 自然语言指令必须先解析为白名单 `PageAction`，禁止直接把自然语言转换成任意点击、任意提交或任意 JS 执行。
- [ ] 所有写入类填充都需要目标槽位校验：若没有找到正确目标、目标不唯一、页面跳转后上下文变化或槽位已被坐席编辑，则停止并要求人工确认。

验收标准：
- 至少覆盖回单、人工确认摘要、缺失信息追问三类半自动填充。
- 每次填充前坐席能预览来源和影响范围。
- 动态填充不绕过状态机、不替代后端 Agent 结论。
- 错误或不可执行场景能给出明确 `disabledReason`，并可降级为人工操作。
- 不允许出现 Page Agent 在未授权情况下发送、结案、提交资料变更或覆盖坐席已编辑内容。

### H3：Page Agent 安全权限与 Harness 失败接管

目标：建立 TicketAgent Page Agent 的安全权限和 Harness 接管机制。即使未来借鉴或接入通用 Page Agent 底层能力，也必须由 TicketAgent 的业务策略层约束动作范围、风险等级、重试次数、超时时间、目标校验和人工接管。该阶段用于体现相对通用 Page Agent 的生产化优势。

开发方案：
- [ ] 定义页面和元素白名单：只允许当前工单页、回单工作区、证据卡片、处理日志、审计卡片和明确可控的业务子窗口；敏感按钮默认黑名单。
- [ ] 定义动作风险分级：极高风险完全禁止，例如发送客户通知、删除、支付、提交拒付、直接结案、改客户资料；中风险必须人工确认，例如填入资料变更草稿、准备拒付建案、准备结案；低风险允许自动执行，例如展开卡片、定位证据、填入未编辑槽位。
- [ ] 增加目标校验：每个动作执行前必须确认页面、工单号、目标组件、目标槽位和当前状态匹配；目标不唯一或目标缺失时停止。
- [ ] 增加 Harness 限制：限制最大操作步数、最大重试次数、单步超时、总耗时和跨页面深度；超限直接进入人工告警/转人工，不继续试错。
- [ ] 增加执行后验证：点击或填充后校验是否到达预期状态、预期组件、预期字段值；验证失败则回滚可回滚的页面草稿或提示人工接管。
- [ ] 增加错误分类：目标未找到、目标不唯一、页面跳转异常、权限不足、状态变化、动作超时、坐席已编辑、风险策略拒绝。
- [ ] 增加 Page Agent 安全审计：记录用户指令、动作计划、策略判定、执行步骤、失败原因、停止原因、人工接管建议。
- [ ] 准备对比说明：通用 Page Agent 强在网页操作泛化，TicketAgent 强在信用卡工单业务策略、风险审计、权限控制和失败接管。

验收标准：
- Page Agent 无法执行未列入白名单的动作。
- 未找到正确目标、目标不唯一、误入子页面、多次失败或超时超限时，必须停止并提示人工接管。
- 极高风险动作被明确禁止，中风险动作必须人工确认，低风险动作可自动执行并留痕。
- 能在答辩中用一个安全案例说明：Page Agent 不会擅自发送、结案、删除或提交高风险业务。

### H4：外部遗留系统页面自动化预研（后置 P2）

目标：仅当真实业务系统没有 API、必须通过浏览器页面处理时，才评估阿里 `page-agent` Ext/MCP 或二开方案。该阶段不是模块 H 当前实现主线；优先 API、Mock API、MCP 或受控 Adapter，页面自动化只作为无 API 场景的最后选择。

预研方案：
- [ ] 梳理外部系统页面清单、登录方式、权限边界、数据敏感级别、可替代 API 和失败接管方式。
- [ ] 评估阿里 `page-agent` 的二开成本：TypeScript monorepo、LLM 配置、Chrome Extension/MCP、本地 hub、DOM 脱敏、动作白名单、供应链和演示稳定性。
- [ ] 设计 TicketAgent Policy Layer：页面白名单、字段脱敏、动作审批、审计记录、人工接管、失败回滚提示。
- [ ] 仅允许 Page Agent 调用受控工具，不允许任意点击、任意 JS 执行或跨系统提交高风险业务动作。
- [ ] 若进入 PoC，先选一个低风险只读页面做查询/定位，不从资金、资料变更或客户通知类操作开始。

验收标准：
- 能明确回答是否值得引入阿里 `page-agent` 作为底层执行引擎。
- PoC 不接触真实敏感客户数据，不修改生产状态。
- 外部页面动作必须有白名单、脱敏、审计、人工确认和失败接管。

### H5：后置能力队列（降优先级）

这些方向保留，但不抢模块 H 的第一主线。

- [ ] 语音入口：当前不开发。信用卡转人工业务以人工客服电话沟通后的发单为主，语音识别、音频上传、ASR 和 App/智能客服自助入口不进入当前业务闭环；如需演示输入来源，仅使用人工发单文本或预置工单内容。
- [ ] Dispatcher Agent：后续作为派单/队列/SLA 能力验证。当前没有 Dispatcher 仍运行良好，是因为系统是 Orchestrator 控制的确定性业务流水线，`workflow_config` 已承担轻量路由，`EscalationAgent` 已承担安全兜底；Dispatcher 未来只负责“派给谁/哪个队列”，不重新分类、不选择业务工具、不覆盖风险结论。
- [ ] A2A-Lite：保留 Agent Card、能力自描述、依赖关系和展示，不做完整协议。
- [ ] MCP：先定义工具接口边界，后续接 MCP Server。
- [ ] LangGraph：当流程分支、状态恢复、回放调试成为主要痛点时再迁移。

模块 H 总体验收标准：
- Page Agent 业务化成为模块 H 的明确第一优先级。
- 语音入口不进入当前业务闭环；Dispatcher Agent 被明确降为后续能力，不再作为 H1/H2 主线。
- Page Agent 能融入当前坐席工作台和现有业务闭环，但不纳入后端 Orchestrator 主编排。
- 系统能讲清相对通用 `page-agent` 的产品优势：不是通用页面自动化，而是信用卡工单场景下具备业务理解、风险审计、权限控制、失败接管、人工确认和可评测指标的受控页面执行层。

## 模块 I：准生产业务沙盘与 Mock 工具生产化（P1 优先）

目标：把当前系统从“AI Demo 工作台”升级为更接近真实信用卡二线工单系统的准生产业务沙盘。模块 I 的重点不是继续堆 Agent 概念，而是补齐业务数据、基础 CRUD、状态流转和可审计 Mock 业务系统，让前端第一眼像业务人员每天使用的处理台，底层 Agent 协作和工具调用默认隐藏到审计明细。

核心判断：
- 当前主库混入过模块 D 测试工单，例如 `T-D-FAIL`、`T-D-MISSING`、`failure_d`、`missing_d`；这些不应出现在业务演示库。
- 当前 `tickets` 表没有独立 `customer_id`，前端曾用 `ticket.id.toUpperCase()` 兜底展示客户号，导致 `FAILURE_D` 这类异常客户号。
- `ai-engine/data/tickets.json` 当前只有 5 条种子工单，无法支撑左侧细颗粒菜单、队列筛选和真实业务观感。
- 当前 Mock 工具偏固定响应，缺少客户、卡片、交易、权益、申请、成功路径矩阵、权限、幂等和业务审计等生产系统特征。
- 评测样本 `evaluation_samples.json` 应继续用于评测和回归，不应直接污染 Demo/业务种子数据库。
- 信息缺失不应默认等同于人工介入；Agent 应先通过受控只读工具跨系统补全字段，确实无法补齐或涉及高风险写入时才进入客户追问/人工接管。

### I1：数据模型与干净业务种子库

目标：建立可反复重置的业务演示数据库，清理测试污染，补齐业务字段，让工单编号、客户号、分类、状态和客户信息都符合银行工单系统口径。

开发方案：
- [ ] 为 `tickets` 增补或迁移核心业务字段：`customer_id`、`category`、`subcategory`、`priority`、`channel`、`assignee`、`department`、`due_at`、`updated_at`、`closed_at`、`final_reply`、`cancel_reason`。
- [ ] 修改 `TicketResponse`、`CreateTicketRequest` 和前端 `Ticket` 类型，前端客户号必须读取 `customerId`，不得再用内部 `id` 兜底展示。
- [ ] 重写 `ai-engine/data/tickets.json` 为 30-50 条可信业务工单，覆盖低/中/高风险、待处理/待补充/待确认/待复核/已升级/已结案等状态。
- [ ] 覆盖信用卡二线常见分类：优惠券补发、机场贵宾厅、道路救援、积分兑换、地址变更、手机号变更、联系人变更、非本人交易、境外交易、调单/拒付、申请进度、额度调整、年费、还款协商、征信异议、投诉升级、跨部门协办。
- [ ] 增加受控 Demo 数据重置能力：只重置本地演示库，不删除评测样本，不写入真实外部系统；重置前在文档中明确会清空本地处理记录。
- [ ] 明确隔离三类数据：业务演示种子 `tickets.json`、评测样本 `evaluation_samples.json`、Mock 业务系统数据（客户/卡片/交易/权益/申请）。

验收标准：
- `/api/tickets` 默认不再出现 `T-D-FAIL`、`T-D-MISSING`、`failure_d`、`missing_d` 等测试工单。
- 每条演示工单都有稳定工单号、客户号、客户姓名、脱敏手机、卡尾号、分类、子分类、优先级、渠道、当前处理人和 SLA 信息。
- 左侧业务菜单主要队列不再大面积为 0，至少 10 个二级分类有可展示数据。
- 前端详情页客户号显示真实 `customerId`。

### I2：基础 CRUD 与业务流转能力

目标：补齐业务系统最小可用功能，让前端不再只是查看和 AI 处理，而是具备工单系统常见的增删改查与状态流转入口。

开发方案：
- [ ] 扩展列表查询参数：工单号、客户号、客户姓名、状态、分类、风险/优先级、处理人、渠道、创建时间范围、SLA 是否超时。
- [ ] 补齐后端接口：`PATCH /api/tickets/{ticket_id}` 编辑工单，`POST /api/tickets/{ticket_id}/assign` 指派，`POST /api/tickets/{ticket_id}/cancel` 作废，`POST /api/tickets/{ticket_id}/reopen` 重开。
- [ ] 保留并强化现有结案接口：实际结案仍必须走 `/api/tickets/{ticket_id}/close`，并写入 `final_reply`、`closed_at` 和处理日志。
- [ ] 增加操作日志模型，记录新建、编辑、指派、作废、重开、启动智能处理、人工确认、填回单、结案等坐席动作。
- [ ] 前端增加业务操作入口：新建工单、编辑基本信息、指派处理人、作废、重开、保存草稿、复核结案。
- [ ] 所有写操作都走明确按钮和 API；智能辅助只能准备内容、定位信息或填草稿，不直接提交业务状态。

验收标准：
- 坐席能在系统内完成新建、查询、查看、编辑、指派、作废/重开和结案的最小闭环。
- 状态流转符合状态机约束，不能从任意状态直接跳到 `closed`。
- 每次业务写操作都有操作日志，能区分坐席动作、系统动作和智能辅助动作。

### I3：Mock 业务系统生产化

目标：把 `MockExecutor` 从“固定返回成功的玩具工具”升级为成功路径更丰富、可审计、可幂等、具备业务规则的模拟核心系统。模块 I 当前演示优先级是“更多成功案例、更少无意义失败和人工介入”，失败/拦截只保留为合规边界和少量必要兜底。

开发方案：
- [ ] 新增 Mock 业务域数据：客户档案、卡片账户、交易流水、优惠券/权益库存、申请单、操作权限、历史工具调用记录。
- [ ] 改造 `mock_executor.py`：工具调用先读取业务域数据并做规则校验，再返回业务结果，不再只套固定 `mock_response`。
- [ ] 扩展信息补全类工具：`customer.lookup`、`customer.profile-query`、`card.account-status-query`、`ticket.history-search`、`knowledge.policy-search`，用于从手机号、卡尾号、姓名、工单内容和历史记录中补齐客户号、卡片、规则和历史上下文。
- [ ] 扩展业务核验类工具：`campaign.eligibility-check`、`coupon.status-query`、`benefit.entitlement-query`、`transaction.detail-query`、`merchant.info-query`、`statement.bill-query`、`application.progress-query`，用于补齐券类型、权益编码、交易流水、商户、账单和申请单状态。
- [ ] 扩展受控执行类工具：`coupon.reissue`、`points.repair-request`、`fee.adjustment-request`、`dispute.case-create`、`customer.update-address`、`notification.send-draft`、`ticket.assign`、`ticket.close-request`；其中写入类工具必须走状态机和人工确认门禁。
- [ ] 增加成功场景矩阵：客户命中、卡片正常、活动达标、券未到账、权益可用、申请单存在、交易可定位、账单可查询、积分可修复、年费可减免申请、争议可建案、资料变更人工确认后可成功执行。
- [ ] 保留少量必要边界提示：重复补发返回已有成功流水和证据，敏感写入进入人工确认，高风险争议进入建案/协办成功而不是直接失败。
- [ ] 增加幂等控制：同一工单/客户/券类型重复补发不能无限成功，重复调用返回已有业务流水和证据。
- [ ] 增加权限门禁：资料变更、拒付、征信异议、高风险交易等只能给出查询或待确认结果，不允许 Mock 直接包装成自动成功。
- [ ] 统一业务返回结构：`success`、`businessStatus`、`businessCode`、`businessMessage`、`operationId`、`evidenceId`、`requiresHuman`、`failureReason`、`nextStep`、`auditPayload`、`durationMs`。
- [ ] 工具审计日志记录请求摘要、脱敏参数、响应摘要、业务流水、证据号、失败原因、耗时和是否需要人工接管。

验收标准：
- 低风险优惠券补发能成功并生成可复用证据号。
- 缺少客户号、券类型、权益编码、申请单号或交易流水号时，系统会先尝试只读查询补齐，而不是直接进入 `pending_info`。
- 至少 10 个以上工具可在演示环境中被调用，并覆盖信息补全、业务核验和受控执行三类场景。
- 重复补发返回幂等结果，不制造多张券。
- 地址变更必须先人工确认；确认通过后可成功更新并生成业务流水。
- 交易争议/拒付优先形成查询结果、建案建议或协办单，不自动结案。
- 工具无法自动完成时不包装为成功，但演示数据应优先覆盖可查询、可补全、可建案、可申请、可复核的正向路径。

### I4：前端业务化与技术细节隐藏

目标：让默认界面面向业务坐席，而不是技术答辩。Agent、Trace、SSE、工具参数和评测指标默认隐藏到审计区或管理页。页面要明确形成“左侧业务工单系统 + 右侧 Agent 辅助区”的双栏叙事，评委一眼能看出业务系统负责处理工单，Agent 负责辅助分析、流转展示、处理日志和回单填充。

开发方案：
- [ ] 调整主布局：业务系统主工作区固定在左侧或中间偏左，承载查询、队列、工单详情、客户/卡片/交易/权益信息、业务按钮和回单编辑；Agent 侧边栏固定在右侧，承载智能建议、状态流转、处理日志、证据、字段补全和技术审计入口。
- [ ] 将 `Agent Copilot` 默认文案改为“智能辅助”或“处理建议”，主操作按钮使用业务动词：生成处理建议、填入回单、定位证据、请求补充、进入复核。
- [ ] 首屏优先展示查询条件、队列列表、工单详情、客户/卡片/交易/权益信息、回单和业务状态流转；处理日志从主表单中移到 Agent 右侧栏，以可展开卡片展示。
- [ ] Agent 右侧栏采用可展开卡片：当前建议、状态流转、处理日志、证据链、字段补全、系统审计。默认展开“当前建议”和“状态流转”，折叠技术审计。
- [ ] 增加动态数据/状态流转展示：启动处理后展示从接单提取、字段补全、业务核验、执行/建案、回单生成、待复核/结案的步骤推进；每一步显示输入、输出、状态变化、证据编号和下一步责任人。
- [ ] 增强点击交互：队列点击切换详情，状态节点点击定位到相关数据卡片，证据点击展开来源，回单按钮点击填入对应子窗口，日志卡片可展开查看前后状态差异。
- [ ] 将 `REST/SSE`、`工具命中率`、`真实链路测评摘要`、`Agent Trace`、工具请求参数等技术信息默认折叠到“系统审计/技术明细”。
- [ ] 新增业务查询和表格列配置：工单号、客户号、客户姓名、分类、优先级、状态、处理人、到期时间、最近处理记录。
- [ ] 详情页按真实工单系统分区：基本信息、客户信息、卡片信息、关联交易/权益/申请、发单内容、回单工作区、附件/备注占位；处理日志和审计明细放到右侧 Agent 区。
- [ ] 移除或重构“字段与风险核验”：不再作为含义模糊的独立表格展示；改为“处理依据/信息核验”卡片，只展示对当前处理有解释价值的字段、来源系统、补全方式、证据编号和风险结论。
- [ ] 增加字段补全展示区：显示“原始工单字段、自动补全字段、补全来源、证据编号、是否仍需人工确认”，让业务人员知道哪些信息来自跨系统查询；若没有补全或核验价值则不展示空表。
- [ ] 将 `pending_info` 文案改为最后兜底状态：只有受控工具无法补齐且无法安全推断时，才展示为“待客户补充”。
- [ ] 保留技术审计入口，但不在普通坐席路径中主动突出 Agent 编排细节。

验收标准：
- 首屏观感接近业务工单系统，不再像独立 AI Demo 页。
- 页面布局能清晰讲出“业务系统在左，Agent 辅助在右”；关闭或折叠 Agent 侧边栏后，业务系统仍可独立查看和处理工单。
- 处理日志、证据链和技术审计不再挤占主业务表单，而是在 Agent 侧边栏中以可展开卡片呈现。
- 点击启动处理后，评委能看到数据从原始工单到字段补全、业务核验、工具执行/建案、回单建议和状态变化的动态流转。
- 普通业务路径不用理解 Agent、Trace、SSE 或 Mock Tool 名称也能完成处理。
- 信息补全结果以业务语言展示为“已从客户/卡片/交易/权益系统核验”，技术工具名默认折叠。
- 技术审计仍可展开查看，满足答辩和排障需要。

### I4a：回单工作区多组件化

目标：把当前“一个文本框”的回单区域升级为真实坐席会使用的回单工作区。回单不只是最终文本，而是由客户回单、内部备注、复核意见、证据附件、后续跟进和填空槽位共同组成，Agent 负责生成建议，坐席负责选择、填充、复核和提交。

开发方案：
- [ ] 将回单区拆成多个子组件/子窗口：客户回单、内部处理意见、复核摘要、客户追问、跟进计划、证据附件、模板选择。
- [ ] 增加模板和填空槽位：如处理结果、证据编号、到账时间、权益有效期、申请节点、交易金额、商户名称、下一步责任人。
- [ ] 增加业务按钮：套用模板、填入 Agent 建议、插入证据编号、生成客户追问、生成内部备注、复制回单、保存草稿、提交复核、结案。
- [ ] 支持不同场景的回单结构：优惠券补发显示补发结果和查收路径；权益查询显示资格、次数和有效期；交易争议显示查询结果、建案/协办建议和材料提示；资料变更显示人工确认和生效说明。
- [ ] 回单工作区与右侧 Agent 建议联动：点击 Agent 侧栏建议时，只填入对应子窗口或槽位，不直接覆盖坐席已编辑内容。
- [ ] 增加草稿状态提示：未生成、已填入、坐席已编辑、待复核、可结案、已提交。

验收标准：
- 回单区不再只有单个文本框，至少包含客户回单、内部备注、复核摘要和证据附件四类组件。
- 坐席能通过多个按钮完成模板套用、建议填充、证据插入、保存草稿和提交复核。
- Agent 填充是可预览、可局部应用、可人工修改的，不直接覆盖确定性证据和状态。
- 不同业务场景的回单结构有明显差异，演示时能体现真实处理深度。

### I5：缺字段自动补全与跨平台只读查询

目标：降低失败和人工介入比例。字段缺失时，Agent 先通过受控工具进行跨系统/跨页面只读查询和证据补全；只有无法补齐、补齐结果冲突、或涉及高风险写入时，才进入客户补充、人工确认或升级。

开发方案：
- [ ] 在 Orchestrator 中增加字段补全阶段，建议位于 Intake 后、Escalation 前：`Intake -> Enrichment -> Escalation -> Resolution -> Notification`。
- [ ] 定义 `FieldEnrichmentResult`：`filledFields`、`unresolvedFields`、`sourceTools`、`evidenceIds`、`confidence`、`conflicts`、`requiresHumanReview`。
- [ ] 定义缺字段补全策略表：`customerId` 可由手机号/卡尾号查客户；`couponType` 可由活动名称/达标记录/优惠券状态查得；`benefitCode` 可由权益名称查得；`applicationNo` 可由客户近期申请查得；`transactionId` 可由日期、金额、商户和卡号查得。
- [ ] 补全工具只允许只读查询，不允许在补全阶段发券、改资料、创建拒付、发送通知或关闭工单。
- [ ] 增加冲突处理：多个客户匹配、活动规则冲突、交易多笔命中、权益不满足、证据不足时，不能强行自动成功，应进入人工复核或客户补充。
- [ ] 将 `pending_info` 从“缺字段默认状态”改为“工具补全失败后的最后状态”；新增或映射 `auto_enriched`、`ready_for_action` 等中间业务状态用于前端表达。
- [ ] Page Agent 跨页面读取仅作为后置补充能力：优先 API/Mock API/MCP，只在没有接口时读取白名单页面和白名单字段；禁止任意点击、任意 JS 执行和跨系统提交。
- [ ] 工具补全过程写入审计日志，记录查询条件、脱敏参数、补全字段、来源系统、证据编号和失败原因。

验收标准：
- 至少覆盖客户号、券类型/活动、权益编码、申请单号、交易流水号五类缺字段自动补全。
- 常见低风险缺字段工单不再直接进入 `pending_info`，而是先形成 `auto_enriched` 或 `ready_for_action` 结果。
- 补全失败时能明确说明“查了哪些系统、为什么仍无法补齐、下一步需要客户或人工补什么”。
- 跨平台/跨页面能力遵循只读优先、白名单、脱敏、审计和人工确认原则。

### I6：数据库从 SQLite 升级到 MySQL/TDSQL

目标：去掉 SQLite 带来的 Demo 感，升级为更贴近企业/银行环境的 MySQL 或云端 TDSQL 数据库。当前项目表结构不复杂，但代码直接依赖 `aiosqlite` 和 SQLite 初始化脚本，因此难度属于中等：不是简单换连接串，需要抽象数据库访问层、处理连接池、迁移脚本、JSON/时间字段、测试库和本地/云端配置。

改造判断：
- MySQL 本地版适合作为第一阶段落地目标，开发和演示稳定性更高。
- TDSQL 可以作为第二阶段配置目标；若兼容 MySQL 协议，应用侧改造基本一致，但要额外处理网络、账号、白名单、SSL、连接池和云端初始化。
- 短期不建议一次性删除 SQLite fallback；可以先让测试/本地 fallback 保留，演示和主运行切到 MySQL/TDSQL。

开发方案：
- [ ] 引入数据库配置：`DB_BACKEND=mysql|tdsql|sqlite`、`DATABASE_URL`、连接池大小、超时、SSL 开关；敏感配置只放 `.env`，示例放 `.env.example`。
- [ ] 选择数据库访问方案：优先评估 SQLAlchemy Async + Alembic，或轻量封装 `aiomysql/asyncmy`；避免业务代码继续散落 SQLite SQL。
- [ ] 将 `models/database.py` 拆成初始化/迁移/连接管理/Repository 层，先覆盖 tickets、ai_results、trace_steps、tool_call_log、evaluations 和新增操作日志。
- [ ] 编写 MySQL DDL 和迁移脚本，处理 `TEXT`/`JSON`、时间字段、索引、唯一约束、状态枚举、外键兼容和字符集 `utf8mb4`。
- [ ] 增加种子数据导入脚本，支持向 MySQL/TDSQL 写入干净业务种子库，并可受控重置演示数据。
- [ ] 更新启动文档：本地 MySQL、云端 TDSQL、SQLite fallback 三种模式的配置方式和注意事项。
- [ ] 增加数据库 smoke：连接检查、建表/迁移、种子导入、CRUD、AI 结果持久化、工具审计、结案回写。

验收标准：
- 后端可通过环境变量切换到 MySQL 或 TDSQL。
- 主演示数据库不再使用 `ai-engine/data/tickets.db`。
- 工单 CRUD、AI 处理结果、Trace、工具审计和结案回写在 MySQL/TDSQL 下均正常。
- SQLite 仅作为本地 fallback 或测试兼容，不作为正式演示口径。

### I7：并行开发组织

建议采用 1 个主 Agent + 2 个子 Agent 并行推进。

主 Agent：架构与集成负责人
- [ ] 维护接口契约、状态机边界、数据隔离原则和最终合并。

子 Agent B：前端/业务体验
- [ ] 负责工单查询、CRUD 入口、业务详情页、字段补全展示、技术细节隐藏、智能辅助业务化文案和前端构建验证。
- [ ] 与子 Agent A 约定字段和接口后并行适配。

模块 I 总体验收标准：
- 系统具备准生产业务沙盘感：数据可信、分类丰富、CRUD 可用、工具调用有真实业务规则。
- 主演示数据库切换到 MySQL 或 TDSQL，SQLite 仅作为 fallback 或测试兼容。
- Mock 工具能表现生产系统常见成功、幂等、权限门禁和必要人工确认场景。
- Agent 能在信息缺失时优先调用受控只读工具自动补全，显著减少直接 `pending_info` 和人工介入场景。
- 工具体系覆盖信息补全、业务核验和受控执行，至少 10 个工具可稳定演示。
- 前端默认面向业务，技术细节可隐藏但可审计。
- 低风险成功、中风险人工确认、高风险升级三条演示链路稳定可复现。
- 修改后至少运行后端 compileall、相关 smoke，以及 `cd frontend && npm run build`。

## 模块 J：评委视角审查、反问演练与答辩收口（P0 答辩前）

目标：站在评委、产品负责人、银行业务方和技术架构评审的视角，对当前 TicketAgent 做一次“挑刺式”审查。模块 J 不优先新增功能，而是围绕“项目是否解决真实问题、Demo 感是否过重、Agent 编排是否合理、技术路线是否可落地、指标是否可信”进行问题发现、针对性系统迭代和答辩话术准备。

核心判断：
- 评委更关心“为什么这个项目有业务价值、为什么不是普通 Chatbot、为什么不是套壳 Demo、为什么能落地到真实业务系统”。
- 当前项目已有 Agent 闭环、状态机、工具审计、评测指标和企业工单壳，但仍需要主动承认 Mock、场景覆盖、真实系统接入、指标口径和生产合规边界。
- 答辩前应先自己扮演评委提问，再把高频质疑转成系统迭代任务、演示证据和回答话术，避免现场被动解释。

### J1：评委视角项目审查清单

目标：从业务价值、产品形态、技术架构、工程完成度、数据与评测、演示可信度六个角度审查当前项目，形成问题清单和优先级。

任务清单：
- [ ] 审查项目解决的问题是否聚焦：明确当前主要解决“进线后智能发单、工具调用、回单复核和人工兜底”，不要把尚未完成的进线前分流、进线中实时陪伴和 13 系统全量接入说成已完成。
- [ ] 审查产品形态是否像真实业务系统：检查默认首页、详情页、菜单、字段、操作按钮、状态流转和审计区是否仍有明显 AI Demo 感。
- [ ] 审查 Agent 编排是否必要且边界清晰：解释五类业务 Agent 为什么存在，为什么没有 Dispatcher，为什么 Escalation 前后两次检查不是冗余。
- [ ] 审查真实业务系统接入程度：区分 Mock Tool、本地准生产沙盘、真实业务 API Adapter、MCP Server 四种阶段，不混淆当前能力和后续规划。
- [ ] 审查指标可信度：明确 40 条真实 LLM 测评是工程回归样本，不是生产真实客户结案率；`closedLoopSuccessRate` 只表述为状态/预期结果匹配率。
- [ ] 审查合规与安全边界：确认高风险、资料变更、交易争议、征信、投诉等场景是否有人工确认、升级、审计和不自动结案边界。
- [ ] 审查演示稳定性：确认演示数据是否干净、关键工单状态是否可复现、失败/升级/成功三条路径是否都有准备。

验收标准：
- 输出一份按 P0/P1/P2 分级的问题清单，P0 必须在答辩前修复或准备明确解释。
- 每个问题都关联到具体页面、接口、Agent、数据或文档，不只写泛泛风险。

### J2：评委可能提问清单

目标：提前准备评委会问的问题，并覆盖产品、业务、技术、数据、落地和竞品对比。

任务清单：
- [ ] 准备业务价值问题：为什么选信用卡工单？痛点是什么？相比人工处理节省在哪里？为什么先做进线后？
- [ ] 准备产品边界问题：当前到底完成到什么程度？哪些是 Demo？哪些是可复用工程能力？哪些明确后置？
- [ ] 准备 Agent 编排问题：为什么需要多个 Agent？为什么不用单 Agent？为什么不用 Dispatcher？为什么不用 LangGraph？为什么 Page Agent 不放进后端编排？

- [ ] 准备可扩展性问题：从 5 类场景扩到 700 类怎么做？规则散落怎么办？`ScenarioRegistry`、工具注册、业务配置和数据集如何演进？
- [ ] 准备竞品/通用 Agent 对比问题：相比普通 Chatbot、RPA、阿里 page-agent、MCP 工具平台，TicketAgent 的差异化是什么？

验收标准：
- 至少形成 30 个评委问题，并按“高概率/中概率/低概率”和“产品/技术/数据/落地/合规”归类。
- 每个高概率问题都有 30 秒版和 2 分钟版回答。

### J3：问题驱动的系统迭代清单

目标：把评委视角发现的问题转化为最小系统改造，不追求大而全，优先补强最容易被质疑的点。

任务清单：
- [ ] 若被质疑“太 Demo”，优先完成模块 I 中的干净业务种子库、真实客户号展示、业务菜单数据覆盖和技术细节默认折叠。
- [ ] 若被质疑“没有真实系统接入”，优先实现一个只读业务系统 API Adapter，例如申请进度查询，并在工具审计中展示来源系统、接口耗时、业务流水和证据编号。
- [ ] 若被质疑“Agent 太多或编排冗余”，补充一张编排职责图和职责边界说明，强调 Orchestrator 确定性推进、Agent 负责业务阶段、Escalation 负责前置/后置安全门禁。
- [ ] 若被质疑“规则散落难扩展”，规划或实现轻量 `ScenarioRegistry`，让字段、必填项、推荐工具、人工确认和通知模板统一来自业务配置。
- [ ] 若被质疑“指标不可信”，在前端和答辩材料中统一改口为“标注样本状态匹配率/工程回归指标”，并展示失败样本如何归因和修复。
- [ ] 若被质疑“LLM 不可控”，补强确定性归一、状态机门禁、证据编号白名单、通知 fallback 和人工复核说明。
- [ ] 若被质疑“无法生产落地”，准备分阶段路线：当前本地沙盘 -> 业务 API Adapter -> MCP 工具服务 -> 真实权限/审计/监控 -> 多系统扩展。

验收标准：
- 每个高优先级质疑都对应一个“已修复、已演示、已解释或明确后置”的处理结论。
- 系统迭代必须优先选择小改动高说服力任务，不在答辩前引入大规模不稳定重构。

### J4：答辩话术稿与演示脚本清单

目标：准备一套可以直接用于答辩的讲解材料，覆盖开场、业务痛点、系统架构、核心演示、指标解释、边界承认和未来规划。

任务清单：
