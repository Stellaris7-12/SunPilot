# 基于多Agent的信用卡智能协同作业系统项目描述

## 1. 项目介绍

信用卡工单系统是客户服务协同的重要平台，承担客户问题处理、业务流转、处理结果沉淀等关键职责。当前信用卡业务场景中，工单处理量大、流程环节多、系统入口分散，业务人员在发单和回单过程中仍需要进行大量人工理解、查询、跳转、复制粘贴和结果填写操作，影响整体处理效率和服务响应速度。

本项目面向信用卡高频工单处理场景，构建一个基于多 Agent 协同的智能工单助手。系统围绕“接单、分类、执行、通知、升级”五类核心业务责任，将传统依赖人工操作的工单处理流程改造为“Agent 自动分析 + 工具自动执行 + 人工复核兜底”的半自动化闭环，最终实现减少人工填写、减少跨系统操作、提升工单处理效率和结果一致性的目标。

项目重点不在于从零建设一个完整大型工单系统，而是在最小可运行工单系统的基础上，验证多 Agent 在信用卡工单业务中的实际提效价值。系统将覆盖权益补发、申请进度查询、资料变更、账单/交易查询、活动资格核验、挂失补卡、额度咨询、年费调整、积分争议、投诉协办等多类场景，避免只围绕少数固定案例硬编码。

## 2. 传统工单业务流程

传统信用卡工单业务通常分为发单和回单两个阶段。

发单阶段主要包括：客户来电或在线咨询、业务员理解客户诉求、核验客户身份、查询客户与卡片信息、选择工单类型、填写工单字段、粘贴通话或问题描述、提交工单并流转至对应处理团队。

回单阶段主要包括：处理人员领取工单、阅读并理解工单内容、判断业务意图、跳转到外部业务系统、执行查询或处理操作、记录处理结果、生成客户回复话术、回填工单并结单。

在该流程中，业务员需要频繁在工单系统、客户信息系统、权益系统、交易系统、商户系统、风控系统等多个系统之间切换。虽然单个操作并不复杂，但在高频场景下会形成明显的时间损耗和操作负担。

## 3. 传统业务痛点

### 3.1 人工理解成本高

客户表达往往具有口语化、碎片化和不完整的特点。同一类问题可能存在多种不同描述方式，业务员需要根据经验判断客户真实意图，并选择对应工单类型或处理路径。对于新员工或复杂问题，理解和判断时间会进一步增加。

### 3.2 字段填写重复且容易出错

工单处理过程中需要反复填写客户姓名、手机号、卡号、交易时间、金额、商户名称、问题描述、处理结果等字段。大量信息来自通话记录、客户资料和外部系统，业务员需要频繁复制粘贴，容易出现漏填、错填和格式不统一等问题。

### 3.3 跨系统跳转影响处理效率

回单时经常需要跳转到多个外部系统完成实际业务操作，例如补发权益、查询交易记录、修改客户地址、核查商户信息等。系统入口分散、操作路径不统一，使得业务员需要记忆大量页面和流程，降低处理效率。

### 3.4 回单话术质量不稳定

不同业务员对处理结果的描述方式不同，可能存在话术不规范、信息遗漏、表达不一致等问题。在金融服务场景下，回单内容需要兼顾准确性、合规性和客户体验，单纯依赖人工撰写会带来质量波动。

### 3.5 自动化过程缺乏可控性

如果仅使用单一大模型直接生成处理结果，容易出现意图误判、字段幻觉、工具调用错误和结果不可追溯等问题。信用卡业务具有较强的合规要求，因此智能化系统不能追求完全无人干预，而需要保留人工复核、过程审计和异常兜底机制。

## 4. 我的方案

本项目提出“业务型多 Agent 协同 + 工具调用 + 人工复核”的智能工单处理方案。系统将工单处理流程拆解为多个职责清晰的业务 Agent，每个 Agent 负责一个明确业务环节，并通过统一编排机制完成端到端协作。

核心流程如下：

```text
原始工单/通话摘要/表单输入
  -> Intake Agent：接单与信息提取
  -> Classifier Agent：分类与优先级判定
  -> Resolution Agent：解决方案与业务执行
  -> Notification Agent：通知、回单与回访
  -> Escalation Agent：异常兜底与人工升级
  -> 人工复核/自动建议结案/转派处理
```

Dispatcher Agent 作为进阶模块，用于后续根据工单类型、紧急程度、人员技能、处理负载或 Agent 能力进行智能派单。

### 4.1 Intake Agent：接单与信息提取

Intake Agent 作为系统“前台”，负责接收来自电话转写、IM、表单、邮件等渠道的用户请求，从中提取工单标题、描述、用户 ID、客户信息、关键业务字段和缺失信息。若信息不足，它可以生成追问或补充字段提示，避免后续执行时参数不完整。

### 4.2 Classifier Agent：分类与优先级判定

Classifier Agent 作为“分诊台”，负责分析工单内容，将其归类为咨询、申请、故障、投诉、争议、资料变更等业务类型，并判断优先级和处理路径。分类策略采用规则 + LLM 的双层方式：规则优先覆盖高频明确场景，LLM 处理口语化、模糊或复杂表达。

### 4.3 Resolution Agent：解决方案与执行

Resolution Agent 作为“一线专家”，负责根据工单类型和抽取字段选择处理方案。它可以查询知识库提供解决建议，也可以调用后端接口、Mock Tool 或未来 MCP 工具完成业务动作，例如补发权益、查询进度、更新资料、查询交易、核验活动资格等。

Resolution Agent 是本项目的重要亮点：系统不只是生成文字建议，而是能够执行可追溯的业务动作，并记录工具请求、工具响应、证据编号和处理耗时。

### 4.4 Notification Agent：通知与回访

Notification Agent 作为“行政助理”，负责在工单状态变化时生成内部通知、客户回单和后续跟进提示。它会根据 Resolution Agent 的执行结果生成标准化回单内容，并在需要时预留满意度回访或后续处理提醒。

### 4.5 Escalation Agent：升级与兜底

Escalation Agent 作为“安全阀”，负责监控字段缺失、工具失败、结果冲突、SLA 即将超时、合规敏感和复杂争议等情况。当系统无法稳定处理时，它会中断自动流程并转交人工、主管或更高等级处理团队。

风险分级不作为独立核心模块，而是 Classifier、Resolution 和 Escalation 共同使用的业务策略。

### 4.6 Dispatcher Agent：智能派单（进阶）

Dispatcher Agent 作为“调度中心”，用于根据工单类型、优先级、紧急程度、工程师技能、当前负载、地理位置或 Agent 能力动态匹配处理人、处理团队或处理 Agent。该模块实现难度较高，当前阶段作为进阶能力，不抢核心闭环。

## 5. 功能规划

结合项目周期和验收要求，系统功能划分为核心功能和进阶功能两部分。核心功能用于保证当前阶段可运行、可演示、可测评；进阶功能用于体现架构创新和后续扩展价值。

### 5.1 核心功能

核心功能聚焦“高频工单的智能处理闭环”，目标是让系统能够完整演示从接单到分类、执行、回单、升级的全过程。

#### 5.1.1 最小工单系统

系统提供基础工单管理能力，包括工单列表、工单详情、工单状态流转、回单草稿、处理记录和结单操作。该模块不追求完整替代企业级工单系统，而是作为智能协同能力的业务承载界面。

#### 5.1.2 多场景工单覆盖

系统不只覆盖少数固定场景，而是围绕信用卡常见工单构建可演示、可评测的场景池。

优先支持：

- 权益/优惠券补发。
- 申请进度查询。
- 地址、手机号、联系人等资料变更。
- 账单/交易查询。
- 活动资格或权益资格核验。

扩展覆盖：

- 分期提前结清咨询。
- 还款协商或延期咨询。
- 卡片挂失、补卡、停卡。
- 额度咨询、临时额度或固定额度调整。
- 年费减免或年费调整。
- 积分到账、积分兑换、积分争议。
- 交易争议、调单、拒付、盗刷疑似场景。
- 征信异议、申请资料补充。
- 商务卡公司资料变更。
- 投诉、催办、跨部门协办。

#### 5.1.3 接单、分类与字段抽取

系统根据工单内容或通话摘要，自动完成接单整理、客户诉求识别、工单类型判断、优先级判定和关键字段抽取。业务员可以直接查看分类依据、字段来源和缺失字段提示。

#### 5.1.4 解决方案与工具调用

系统根据分类结果和字段信息，自动匹配对应业务工具。Demo 阶段将补券、查交易、查进度、改地址、查权益等动作封装为 Mock Tool API，模拟真实业务系统调用流程。后续可以将 Mock Tool 替换为 MCP Server、真实内部接口或审批流接口。

#### 5.1.5 自动回单与通知

系统根据工单内容、工具调用结果和业务规则生成规范化回单话术。话术内容需要包含处理结果、证据编号、必要说明和客户可理解的回复表达，业务员可一键采纳或编辑后提交。

#### 5.1.6 人工复核与升级兜底

系统在关键动作前保留人工确认机制。Agent 负责完成重复性分析、工具执行和草稿生成，业务员负责最终复核、确认和结单。对于字段缺失、工具失败、复杂争议或合规敏感场景，Escalation Agent 会转人工或升级处理。

#### 5.1.7 Agent执行轨迹与审计证据

系统记录每次处理中的分类结果、字段抽取结果、工具调用参数、工具返回结果、证据编号、回单话术和人工修改记录，形成可追溯的 Agent 执行链路，为问题定位和后续评测提供依据。

### 5.2 进阶功能

进阶功能用于体现项目的前沿性和扩展性，可以根据开发进度选择实现其中 1 到 2 项作为亮点。

#### 5.2.1 A2A-lite 多Agent协同架构

项目可借鉴 A2A 架构思想，实现轻量级 Agent 协同机制。每个 Agent 通过能力卡片描述自身职责、输入输出、依赖关系和是否需要人工确认，由主控编排器根据当前工单场景选择处理链路。

当前只保留 Agent Card、能力自描述和前端展示，不做完整 A2A 协议、跨进程通信或复杂协商。

#### 5.2.2 MCP思想下的工具注册中心

项目可借鉴 MCP 的工具暴露思想，将补券、查交易、改地址、查客户权益等业务动作封装为统一工具。每个工具定义名称、描述、输入参数、返回结构和审计字段，供 Resolution Agent 统一调用。

在 Demo 阶段，工具可以使用 Mock API 模拟；在真实落地阶段，工具可以替换为内部系统接口、数据库查询、审批流接口或 MCP Server。这样既能保证演示稳定，也能体现后续接入真实业务系统的可行性。

#### 5.2.3 Page Agent 页面助手与遗留系统自动化

Page Agent 分三阶段引入：

1. **前端页面助手**：在 `frontend/src/views/TicketDetailView.vue` 增加“页面助手”入口，只操作当前工单详情页，例如把 AI 回单草稿填入回单框、检查缺失字段、打开工具面板并定位补券工具、滚动到审核区域。
2. **发单/回单表单自动填充**：等动态工单表单更完整后，把 Intake Agent 的抽取结果转成页面填充动作，减少人工复制粘贴。
3. **外部遗留系统自动化**：如果某些行内系统没有 API，才考虑 Page Agent Ext / MCP 方案操作浏览器页面。该能力属于高风险扩展，必须具备白名单、脱敏、审计和人工确认。

当前阶段优先考虑第一阶段“页面助手”，不把外部系统自动点击作为核心交付。

#### 5.2.4 Agent评测面板

系统可以设计小型 Agent 评测面板，对每个 Agent 节点的效果进行统计和分析，包括字段抽取完整率、意图准确率、工单类型准确率、工具选择正确率、回单要点覆盖率、人工修改率和升级判断合理性。

该功能可以帮助定位多 Agent 链路中的薄弱环节。例如，如果字段抽取完整率低，就优化 Intake Agent；如果工单类型错误率高，就优化 Classifier Agent；如果工具选择错误率高，就优化 Resolution Agent；如果回单人工修改率高，就优化 Notification Agent。

#### 5.2.5 多场景扩展与动态编排

在核心场景稳定后，系统可以扩展更多信用卡业务场景。通过 A2A-lite 能力注册、工具注册中心和轻量 workflow_config，主控编排器可以根据不同工单类型选择处理链路，实现从单一流程向多场景智能协同平台演进。

## 6. 为什么引入多Agent和Page Agent

信用卡工单处理不是单纯的问答任务，而是一个包含接单、分类、执行、通知、升级和复核的多步骤业务流程。单个大模型虽然可以生成文本，但难以稳定完成复杂链路中的职责划分、状态追踪和错误定位。

引入多 Agent 后，可以将复杂流程拆解为多个可解释、可评估、可替换的节点。每个 Agent 只负责相对明确的业务责任，从而降低单点复杂度，提高系统可维护性和可控性。同时，每个 Agent 的输入输出都可以被记录下来，便于后续问题追踪和效果评测。

Page Agent 主要用于减少页面操作成本和适配无 API 的遗留系统。在当前阶段，它更适合先作为当前工单详情页的“页面助手”，帮助填入回单草稿、检查缺失字段、定位工具区域。等动态表单和安全边界完善后，再考虑表单自动填充和外部系统页面自动化。

## 7. 业务优化效果

本项目预期从以下几个方面提升工单业务效率。

### 7.1 缩短单笔工单处理时间

传统流程中，业务员需要手动阅读工单、查询系统、填写字段和撰写回单，单笔工单可能耗时 1 到 2 分钟。引入智能协同后，高频标准场景可由 Agent 自动完成大部分前置分析和工具调用，业务员只需复核确认，目标将处理时间压缩至秒级或分钟级。

### 7.2 减少人工填写和复制粘贴

系统可以自动从工单文本或通话摘要中提取关键字段，并生成结构化结果。业务员不再需要在多个页面之间反复复制粘贴，从而降低重复劳动和错填风险。

### 7.3 降低跨系统操作成本

通过 Resolution Agent 和工具注册中心，系统可以把复杂的跨系统操作封装成统一工具能力。业务员不需要记忆不同系统入口和操作路径，只需在工单页面完成确认。

### 7.4 提升回单话术规范性

Notification Agent 可以根据统一模板和业务规则生成处理结果说明，使回单表达更加规范、完整和一致，降低由于个人经验差异导致的服务质量波动。

### 7.5 增强流程可追溯性

系统会记录每个 Agent 的输入输出、工具调用参数、处理结果、证据编号和人工确认记录，形成完整执行链路。这不仅便于问题排查，也有助于后续模型评测和流程优化。

## 8. 项目创新点

### 8.1 从“生成文本”升级为“完成业务动作”

传统 AI 客服或工单助手多停留在摘要生成、话术推荐等文本层面。本项目进一步引入工具调用能力，使 Agent 能够在识别诉求后执行模拟业务操作，形成“理解-执行-回填”的闭环。

### 8.2 业务型多Agent职责拆分

项目将工单处理拆分为 Intake、Classifier、Resolution、Notification、Escalation 等业务 Agent。相比单一大模型端到端生成结果，该架构更便于调试、追踪、评测和向业务方解释。

### 8.3 A2A-lite能力发现机制

项目借鉴 A2A 的 Agent 能力发现和协作思想，通过 Agent 能力卡片描述每个 Agent 的职责、输入输出和依赖关系，使系统具备更好的扩展性，也更容易解释多 Agent 之间如何协同工作。

### 8.4 MCP工具化业务能力

项目借鉴 MCP 的工具抽象思想，将补券、查交易、改地址、查权益等业务动作封装为统一工具，由 Resolution Agent 按需调用。该设计将“业务操作”从页面点击中抽象出来，使 Agent 不只生成文本，还能真正完成业务动作。

### 8.5 人工复核机制适配金融场景

信用卡业务对准确性和合规性要求较高，因此系统不追求完全无人值守，而是在关键节点保留人工确认。Agent 负责完成重复性和规则性工作，业务员负责最终判断和风险把关。

### 8.6 Agent评测体系辅助持续优化

项目可以设计小型 Agent 评测面板，对字段抽取完整率、意图准确率、工单类型准确率、工具选择正确率、回单采纳率和人工升级合理性进行统计。该机制可以帮助定位多 Agent 链路中的薄弱节点，为后续优化提供依据。

### 8.7 Page Agent探索页面操作自动化

Page Agent 先作为当前工单详情页的页面助手，再逐步扩展到动态表单自动填充。对于无法直接接入接口的内部系统，未来可以在严格白名单、脱敏、审计和人工确认的前提下探索网页自动操作。

## 9. 项目意义

本项目的业务意义在于提升信用卡工单处理效率，减少业务员在重复填写、跨系统跳转和话术撰写上的时间消耗，使其能够将更多精力投入复杂问题判断和客户服务质量提升。

本项目的技术意义在于探索大语言模型从“内容生成工具”向“业务流程执行助手”的转变。通过多 Agent 协同、工具调用、人机协同和过程审计，系统能够更贴近真实企业业务流程，而不是停留在简单问答或文本生成层面。

本项目的落地意义在于采用“小而精”的实现策略，在有限时间内聚焦高频、标准、可验证的工单场景，快速构建可运行 Demo，并通过处理时长、字段填写量、回单采纳率、工具命中率等指标证明智能化改造效果。

最终，本项目希望实现的不是替代业务员，而是为业务员提供一个可靠的智能协同助手：由 Agent 完成重复性、流程性、标准化工作，由人工完成复核、判断和风险把控，从而推动信用卡工单系统向智能化、高效化和可持续优化方向演进。
