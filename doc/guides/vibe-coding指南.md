# Vibe Coding 指南

这份指南面向“用 AI 会话持续推进项目开发”的工作方式。它不是某个项目的详细排期，而是一套通用操作流程：如何拆模块、开会话、并行开发、测试、迭代、维护 `AGENTS.md` 和 planning 文件。

核心原则：

```text
用小闭环推进，用清晰边界并行，用测试会话验收，用文档保持上下文。
```

## 一、总体工作流

每一轮 vibe coding 都按下面的顺序走：

1. 明确本轮目标和约束。
2. 拆分模块和优先级。
3. 冻结当前阶段的接口和数据契约。
4. 开发一个可验收的小闭环。
5. 做单模块测试。
6. 做跨模块集成测试。
7. 更新文档、planning 和当前事实。
8. 提交一次稳定快照。
9. 再进入下一轮迭代。

不要一轮里同时追求“功能完整、架构高级、UI 完美、数据集丰富”。每轮只选一个主目标，其他目标服务它。

## 二、模块推进顺序

通用顺序是：

```text
核心闭环
  -> 数据流和接口
  -> 外部依赖 Mock
  -> 单模块能力
  -> 集成链路
  -> 数据集和测评
  -> 用户体验
  -> 进阶能力
```

### 1. 先核心闭环，再增强体验

先证明系统能把主流程跑完，再优化页面、文案、交互和演示效果。

例如 TicketAgent 中，优先级应是：

```text
工单输入 -> 分类 -> 抽取 -> 风控 -> 工具处理 -> 回单 -> 结单
```

而不是先做语音、PageAgent 或复杂动画。

### 2. 先确定数据流，再优化 UI

UI 依赖稳定的数据结构。接口还在变时，不要让前端会话大规模美化页面。

推荐先冻结：

- API 请求参数。
- API 返回字段。
- 状态枚举。
- 错误格式。
- 前端类型定义。

### 3. 先 Mock 外部依赖，再接真实服务

真实外部服务通常带来账号、网络、权限、稳定性问题。先用 Mock 证明业务路径正确，再替换真实能力。

适合 Mock 的内容：

- 第三方 API。
- 业务系统接口。
- 语音识别。
- 支付、交易、工单平台操作。
- 大模型不可控输出的部分兜底。

### 4. 先小数据集验证，再扩样本

不要一开始追求大数据集。先做 10-20 条高质量样本，覆盖主路径和失败路径，再逐步扩展。

每条样本至少包含：

- 输入。
- 期望输出。
- 关键字段。
- 正确分类。
- 预期处理动作。
- 风险等级或异常标签。

### 5. 先规则兜底，再引入复杂 Agent 或框架

复杂 Agent、图编排、自动规划框架都很诱人，但它们会放大调试成本。

推荐顺序：

```text
明确规则
  -> 简单 Agent
  -> 配置化流程
  -> 多 Agent 协作
  -> 图编排/复杂框架
```

## 三、会话类型

把不同目标放进不同会话，能减少上下文污染。

| 会话类型 | 主要职责 | 是否改代码 |
|----------|----------|------------|
| 主控会话 | 定目标、拆模块、定接口、合并和验收 | 少量 |
| 模块开发会话 | 只开发一个明确模块 | 是 |
| 测试会话 | 只验证、复现、定位问题 | 尽量不改 |
| 文档会话 | 写说明、演示稿、交接文档 | 是 |
| 调研会话 | 比较方案、评估风险、给建议 | 否 |
| Code Review 会话 | 审查风险、缺陷、测试缺口 | 否或少量 |

### 主控会话

主控会话负责“方向感”：

- 当前做什么。
- 不做什么。
- 接口是否冻结。
- 哪些模块可以并行。
- 哪些改动可以合并。
- 是否达到验收标准。

主控会话不适合长时间陷入某个 bug 的细节。

### 模块开发会话

模块开发会话只处理一个范围清楚的任务。

好的模块任务：

```text
实现 evaluation_samples.json 的读取和指标计算，不改前端 UI。
```

不好的模块任务：

```text
顺便把测评、前端、Agent、数据库都优化一下。
```

### 测试会话

测试会话的目标是证明“现在到底能不能用”。

它应该输出：

- 运行了什么命令。
- 哪些通过。
- 哪些失败。
- 失败的复现步骤。
- 初步归因。
- 是否建议回到开发会话修复。

### 文档会话

文档会话负责把系统状态讲清楚。

适合输出：

- README。
- 启动指南。
- Demo 讲解。
- 接口说明。
- 交接说明。
- 规划文件。

文档会话不要随手改核心代码。

## 四、什么时候新开会话

适合新开会话的情况：

- 当前会话上下文太长，目标开始混乱。
- 一个模块已开发完成，需要独立测试。
- 从开发切换到测试、重构、文档或汇报。
- 出现连续修复失败，需要换视角诊断。
- 准备并行开发另一个相对独立模块。
- 需要 Code Review，而不是继续写代码。
- 需要让 AI 忘掉刚才的错误假设，重新从事实出发。

新会话的价值是“切换角色”和“清理上下文”，不是单纯开得越多越好。

## 五、什么时候不要新开会话

不要新开会话的情况：

- 同一个 bug 仍在连续定位中，关键上下文都在当前会话。
- 模块边界还没想清楚。
- API 或数据结构还没冻结。
- 多个会话会同时改同一个核心文件。
- 你还没有明确新会话的输入、输出和验收标准。

尤其要避免多个会话同时修改：

- 核心编排文件。
- 数据库 schema。
- 全局类型定义。
- 路由和 API 契约。
- 共享状态管理。

这些文件适合由主控会话或单一开发会话串行处理。

## 六、并行开发规则

并行开发的前提是边界清楚、接口稳定。

### 适合并行的模块

- 数据集构建。
- 文档整理。
- Demo 讲解。
- 前端静态展示。
- 独立评测脚本。
- 单个工具 Mock。
- 单个页面或组件。

### 谨慎并行的模块

- 后端 API 和前端类型。
- Agent 输出和测评脚本。
- 数据库字段和前端状态展示。
- 配置文件和编排逻辑。

这些模块可以并行，但必须先由主控会话冻结契约。

### 不建议并行的模块

- 核心业务流编排。
- 数据库迁移。
- 全局状态机。
- 认证/权限/安全相关逻辑。
- 大规模重构。

这类任务并行会让冲突和回归风险迅速上升。

## 七、测试节奏

测试不要只放在最后。每个模块都有自己的验收层级。

### 1. 单元级测试

适合验证：

- 函数。
- 数据结构。
- 字段转换。
- 简单规则。
- 组件 props。

目标：局部逻辑正确。

### 2. 模块级测试

适合验证：

- 一个 API。
- 一个 Agent。
- 一个前端组件。
- 一个数据处理脚本。
- 一个工具调用。

目标：模块输入输出正确。

### 3. 集成级测试

适合验证：

- 前后端接口联通。
- 多 Agent 串联。
- 数据库读写。
- 工具调用和结果展示。
- 完整业务流。

目标：多个模块合在一起能跑通。

### 4. 回归级测试

每轮开发后至少跑主路径和关键失败路径。

推荐覆盖：

- 主路径成功。
- 缺字段。
- 外部工具失败。
- 高风险拦截。
- 用户取消或拒绝。
- 重复执行。

### 5. 演示级测试

从最终用户视角完整走一遍。

检查：

- 是否能启动。
- 是否能按脚本完成演示。
- 页面是否有明显错位。
- 结果是否能被非技术人员看懂。
- 失败时是否有合理提示。

## 八、每个模块完成后的通用验收

每个模块完成后都问这 7 个问题：

1. 输入是什么？
2. 输出是什么？
3. 成功路径是否跑通？
4. 失败路径是否可解释？
5. 是否影响其他模块？
6. 是否更新了文档或 planning？
7. 是否可以提交一个稳定快照？

如果答不上来，说明模块还没有真正完成。

## 九、迭代方法

### 每轮只定义一个主目标

好的迭代目标：

```text
让低风险工单从输入到自动回单完整跑通。
```

不好的迭代目标：

```text
把系统整体优化一下。
```

### 复杂功能先做最小可用版

不要一开始追求完整形态。

例子：

| 复杂目标 | 最小可用版 |
|----------|------------|
| 语音处理 | 先用预置转写文本模拟 |
| 多系统调用 | 先用 Mock Tool |
| 自动评测 | 先做 20 条样本 |
| Agent 编排 | 先手写主流程，再配置化 |
| 插件化 | 先做独立工作台 |

### 每次失败都要改变策略

不要重复同一个失败动作。

记录：

- 失败现象。
- 尝试过什么。
- 失败原因猜测。
- 下一次换什么方法。

### 每轮结束必须留下可运行版本

每轮结束前至少做到：

- 代码能启动。
- 主路径不坏。
- 重要变更有说明。
- planning 或进度记录已更新。
- Git 有一个可回退点。

## 十、如何维护 planning 文件

planning 文件是跨会话的长期记忆。

建议分工：

| 文件 | 作用 |
|------|------|
| `task_plan.md` | 当前阶段目标、模块计划、验收标准 |
| `findings.md` | 关键发现、决策、风险、取舍 |
| `progress.md` | 每次会话做了什么、改了什么、测了什么 |

更新时机：

- 开始新阶段前。
- 模块完成后。
- 重大决策后。
- 测试失败后。
- 提交前或提交后。
- 新会话接手前。

不要把所有细节都塞进 `task_plan.md`。计划文件应该清楚，不应该变成日志堆。

## 十一、如何写 AGENTS.md

`AGENTS.md` 是给 AI 开发会话看的项目级规则。它应该帮助 AI 不跑偏。

### 应该写什么

- 项目目标。
- 当前阶段优先级。
- 目录职责。
- 核心接口和数据契约。
- 哪些文件改动要谨慎。
- 禁止做的事。
- 常用启动和测试命令。
- 文档更新要求。
- 提交和安全规则。

### 不应该写什么

- 过时的详细排期。
- 频繁变化的临时任务。
- 大段业务背景。
- 模糊口号。
- 与仓库事实不一致的命令。

### AGENTS.md 简版模板

```md
# AGENTS.md

## 项目目标

一句话说明项目要解决什么问题。

## 当前阶段优先级

- P0：当前必须优先完成的核心闭环。
- P1：增强能力和体验。
- P2：有余力再做的进阶能力。

## 目录职责

- `src/`：核心代码。
- `frontend/`：前端。
- `doc/`：文档。
- `doc/planning/`：计划、发现、进度。

## 开发约定

- 修改接口时同步类型定义和调用方。
- 修改数据库时只做兼容迁移。
- 修改核心编排前先阅读相关模块。
- 不做无关重构。

## 测试命令

列出后端、前端、集成测试命令。

## 禁止事项

- 不提交密钥。
- 不提交构建产物。
- 不删除重要文件。
- 不修改项目外文件。
```

## 十二、Prompt 模板

### 1. 新会话开场 Prompt

```text
你现在接手这个项目的一个新会话。

请先只读：
1. AGENTS.md
2. doc/planning/task_plan.md
3. doc/planning/findings.md
4. doc/planning/progress.md
5. 与本任务相关的代码文件

本会话目标：
[写清楚一个目标]

范围限制：
[写清楚哪些文件/模块可以改，哪些不要碰]

验收标准：
[写清楚如何判断完成]

请先复述你理解的目标和计划，再开始修改。
```

### 2. 模块开发 Prompt

```text
请只开发 [模块名]。

输入：
[模块输入]

输出：
[模块输出]

不要修改：
[禁止修改的模块]

完成标准：
1. [标准一]
2. [标准二]
3. [标准三]

完成后请说明改了哪些文件、如何测试、还有什么风险。
```

### 3. 测试会话 Prompt

```text
请作为测试会话，不要先改代码。

目标：验证 [功能/模块] 是否可用。

请执行：
1. 阅读相关接口和文档。
2. 运行必要测试。
3. 记录通过项和失败项。
4. 对失败项给出复现步骤和可能原因。

只有当问题非常明确且修复很小的时候，才提出补丁建议。
```

### 4. Code Review Prompt

```text
请对当前改动做 code review。

优先关注：
- 业务逻辑错误。
- 接口契约破坏。
- 状态流转问题。
- 数据库兼容性。
- 缺失测试。
- 安全和敏感信息。

请按严重程度列出问题，引用具体文件和位置。
```

### 5. 文档会话 Prompt

```text
请只更新文档，不改源码。

目标读者：
[开发者/产品经理/客户/领导/测试人员]

文档目标：
[启动指南/演示讲解/接口说明/交接文档]

要求：
- 简洁。
- 面向读者。
- 不写过多无关技术细节。
- 路径和命令必须与仓库一致。
```

## 十三、每轮迭代检查清单

开始前：

- [ ] 本轮目标是否只有一个？
- [ ] 范围是否清楚？
- [ ] 是否知道哪些文件不能碰？
- [ ] 是否已有验收标准？
- [ ] 是否需要先冻结接口？

开发中：

- [ ] 是否只改相关模块？
- [ ] 是否避免无关重构？
- [ ] 是否记录了重要决策？
- [ ] 是否有失败记录？

测试前：

- [ ] 单模块是否可运行？
- [ ] 测试数据是否准备好？
- [ ] 失败路径是否覆盖？

提交前：

- [ ] 工作区是否只包含本轮相关改动？
- [ ] 是否没有提交密钥、数据库、依赖目录、构建产物？
- [ ] 是否更新了 planning 或文档？
- [ ] 是否有一句清楚的提交说明？

新会话交接前：

- [ ] 当前事实是否写入 `progress.md`？
- [ ] 剩余问题是否明确？
- [ ] 下一步是否有输入、输出和验收标准？

## 十四、一个实用节奏

推荐节奏：

```text
主控会话定目标
  -> 模块开发会话实现
  -> 测试会话验收
  -> 主控会话合并判断
  -> 文档会话更新说明
  -> 提交稳定快照
```

如果时间紧，就缩短为：

```text
目标明确 -> 小步实现 -> 立即测试 -> 更新进度 -> 提交
```

真正高效的 vibe coding，不是让 AI 一口气写很多代码，而是让每一轮都留下一个更稳定、更清楚、更可继续的项目状态。
