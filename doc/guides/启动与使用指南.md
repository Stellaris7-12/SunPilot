## 二、快速启动

> 推荐方式：**只使用项目根目录 `.venv`**，并由 `uv` 统一管理 Python 依赖。  

### 1. 进入项目目录（PowerShell）

```powershell
cd C:\Users\heyunhui\PyProjects\TicketAgent
```

### 2. 初始化 Python 环境

首次执行：

```powershell
uv sync
```

后续如果只是启动项目，不需要手动激活 `.venv`。所有后端命令都用 `uv run` 执行。

如果需要检查后端依赖是否可用：

```powershell
uv run python -c "import fastapi, openai, aiosqlite; print('backend deps ok')"
```

### 3. 配置 LLM API Key

PowerShell 当前终端临时配置：

```powershell
$env:DEEPSEEK_API_KEY="sk-your-deepseek-api-key"
```

可选配置（有默认值，按需设置）：

```powershell
$env:LLM_BASE_URL="https://api.deepseek.com/v1"
$env:LLM_MODEL="deepseek-chat"
$env:LLM_TIMEOUT="30"
$env:LLM_MAX_TOKENS="2000"
$env:LLM_TEMPERATURE="0.1"
$env:PORT="8000"
```

### 4. 启动后端

从项目根目录执行：

```powershell
uv run python -m uvicorn main:app --app-dir ai-engine --reload --port 8000
```

启动成功后输出：
```
INFO:     Started server process
INFO:     Waiting for application startup.
INFO:     Starting Credit Card Multi-Agent System...
INFO:     Database initialized. 5 agents registered.
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8000
```

### 5. 安装前端依赖并启动

新开一个终端：

```powershell
cd C:\Users\heyunhui\PyProjects\TicketAgent\frontend
# 初次需安装npm
npm.cmd install
# 后续可直接运行
npm.cmd run dev
```

> PowerShell 里直接执行 `npm run dev` 可能触发 `npm.ps1` 执行策略报错，所以推荐使用 `npm.cmd`。

浏览器打开 **http://localhost:5173**，系统会进入企业工单系统首页；也可以直接访问 **http://localhost:5173/tickets**。

当前默认入口已经切换为模块 G+ 的企业工单系统壳。旧版坐席工作台仍保留为 fallback，可访问 **http://localhost:5173/legacy/tickets**。

### 6. Bash / Git Bash 等价命令

如果使用 Bash / Git Bash：


## 三、关闭服务

### 方法一：Ctrl+C（正常关闭）

在运行前后端的终端窗口中，按 **`Ctrl + C`** 即可停止该服务。

> **Windows CMD 注意**：如果 Ctrl+C 无响应（通常因为存在活跃的 SSE 长连接），按 **`Ctrl + Break`**（Pause 键），或使用方法二强制终止。

### 方法二：强制终止端口（Ctrl+C 无效时）

```bash
# 查看占用 8000 端口的进程 PID
netstat -ano | findstr :8000

# 记下输出最后一列的 PID，强制终止（替换为实际 PID）
taskkill /PID 12345 /F

# 或一步到位
for /f "tokens=5" %a in ('netstat -ano ^| findstr :8000 ^| findstr LISTENING') do taskkill /PID %a /F
```

### 恢复初始状态

如果想重置所有工单到初始状态，需要删除本地 SQLite 运行库文件并重启后端。该操作会清空本地处理记录，执行前请先确认。

PowerShell 命令如下：

```powershell
Remove-Item .\ai-engine\data\tickets.db -ErrorAction SilentlyContinue
Remove-Item .\ai-engine\data\tickets.db-shm -ErrorAction SilentlyContinue
Remove-Item .\ai-engine\data\tickets.db-wal -ErrorAction SilentlyContinue
```

### 关于虚拟环境

使用推荐的 `uv run ...` 启动方式时，不需要手动激活或停用虚拟环境。`uv` 会自动使用项目根目录 `.venv`。

## 四、使用流程

### 企业工单系统壳概览

模块 G+ 完成后，前端默认更接近真实银行/客服二线工单系统：主系统负责菜单、标签页、详情表单、处理日志、回单和状态流转；Agent Copilot 以右侧推入式插件栏接入，只做辅助建议和页面定位。

```
┌──────────────┬──────────────────────────────────────────────┬──────────────────┐
│ 业务菜单树     │ 主工单系统工作区                                │ Agent Copilot    │
│              │                                              │                  │
│ 受理队列       │ 顶部多标签：工作首页 / 已打开工单                  │ 当前建议          │
│ 权益与活动     │ 首页：状态池、优先队列、真实链路评测摘要            │ 证据/缺失         │
│ 客户资料       │                                              │ 生产约束          │
│ 交易与风险     │ 详情：基本信息、客户与卡片信息、发单内容            │                  │
│ 申请与账务     │      字段核验、证据链、处理日志、回单内容           │ 可隐藏/展开       │
│ 人工协办       │ 底部：技术审计 / Agent 执行明细（默认折叠）         │ 不遮挡主表单      │
└──────────────┴──────────────────────────────────────────────┴──────────────────┘
```

### 1. 企业工单首页接单

1. 打开 **http://localhost:5173/tickets**。
2. 左侧按真实信用卡二线视角查看菜单：受理队列、权益与活动、客户资料、交易与风险、申请与账务、人工协办。
3. 可点击二级队列，例如优惠券补发、餐饮券/满减券、地址变更、非本人交易、调单/拒付、申请进度、投诉升级等。
4. 中间“当前优先队列”展示需要坐席关注的工单，包含工单号、标题、场景、风险、建议动作和当前状态。
5. 点击某一张工单，进入案件处理台。
6. 也可以直接访问可见工单编号，例如 **http://localhost:5173/tickets/T20260715001**；前端会自动解析到内部工单 id。

### 2. 案件处理台

进入工单详情后，优先看主系统区域：

- **业务场景**：如权益/优惠券、资料变更、交易/争议。
- **当前状态**：待处理、待补充、待确认、待复核、已升级、已结案。
- **发单内容**：人工客服转办后的原始工单描述。
- **下一负责人**：客户补充、坐席/复核岗、人工团队或系统继续处理。
- **证据编号**：工具执行后生成的业务证据编号。
- **回单草稿**：坐席复核后才能提交结案。

中间主区域按坐席阅读顺序展示：

1. 基本信息：工单编号、状态、场景、紧急程度、创建时间、下一责任人、SLA。
2. 客户与卡片信息：客户号、手机号、卡号后四位、卡产品和权益状态。
3. 发单内容：人工客服转办后的原始描述。
4. 字段与风险核验：已提取字段、缺失字段和风险标签。
5. 证据链：工具审计和业务证据编号。
6. 处理日志：原系统操作记录 + Agent 摘要。
7. 回单内容：AI 草稿进入主系统回单框，由坐席复核。
8. 技术审计 / Agent 执行明细：默认折叠，用于调试和答辩追问。

右侧 **Agent Copilot** 提供当前工单的辅助动作：

- 启动 AI 辅助 / 重新生成建议。
- 填入回单草稿。
- 定位证据。
- 查看缺失字段。
- 打开技术审计。
- 进入人工确认。
- 进入回单复核。

Copilot 不直接保存、结案、转派或覆盖主系统状态。实际结案必须点击主系统 **“复核并结案”** 按钮，且前端会要求工单处于 `pending_human_review` 并存在 `closureSuggestion.canClose=true`。

### 3. 评测摘要

首页底部展示评测摘要，读取 `GET /api/evaluation/metrics`：

- 状态匹配率：工程含义是“状态/预期结果匹配率”，不是实际客户结案率。
- 工具命中率：Resolution 工具选择与参数结果。
- 字段完整率：Intake 关键字段抽取结果。
- 平均耗时：真实链路平均处理耗时。
- 样本数：当前评测样本数量，模块 F 完成后为 40 条。

### Demo 场景 1：优惠券补发（低风险，工具执行型）

1. 在工作池中选择 **“活动达标未收到餐饮优惠券”**。
2. 基本信息区显示场景为“权益/优惠券”，坐席关注点为达标原因、补发结果和证据编号。
3. 点击右侧 **“启动 AI 辅助”**。
4. 处理日志和技术审计依次记录：接单提取 → 业务分诊 → 风险拦截 → 执行处理 → 回单生成。
5. 证据链区域展示 `coupon.reissue`、业务结果、证据编号和下一步建议。
6. 点击 **“填入回单草稿”**，检查客户标准回单。
7. 工单进入待复核且结案建议允许后，点击主系统 **“复核并结案”**。

### Demo 场景 2：地址修改（中风险，需人工确认）

1. 在工作池中选择 **“客户要求修改账单寄送地址”**。
2. 基本信息区显示场景为“资料变更”，风险为中风险。
3. 点击 **“启动 AI 辅助”**。
4. 系统识别身份核验、地址字段和敏感操作要求，流程进入待确认。
5. 弹出 **“人工确认敏感操作”** 对话框；刷新后也可通过主页面或 Copilot 的 **“进入人工确认”** 重新打开。
6. 坐席检查字段和客户身份后，点击 **“确认执行”**；如不认可，点击 **“转人工处理”**。
7. 处理完成后复核回单并结案。

### Demo 场景 3：交易争议/盗刷疑似（高风险，自动升级）

1. 在工作池中选择 **“客户反馈199元交易非本人消费”** 或类似交易争议工单。
2. 基本信息区显示场景为“交易/争议”，风险为高风险。
3. 点击 **“启动 AI 辅助”**。
4. 系统保留客户诉求、风险判断和人工复核摘要，不直接给出自动业务结论。
5. 页面显示升级原因、接管建议和后续人工归属。
6. 坐席可展开“技术审计 / Agent 执行明细”查看底层执行链路。

## 七、架构设计要点

### 5个业务Agent协作流水线

```
工单输入 → Classifier Agent（分类与优先级判定）
         → Intake Agent（接单与信息提取）
         → Escalation Agent（升级与兜底）
         → Resolution Agent（解决方案与执行）
         → Notification Agent（通知与回单）
         → 人工终审 → 结单
```

### 三级风险路由

| 风险等级 | 处理方式 | 示例场景 |
|----------|---------|---------|
| **LOW** | 全自动5Agent管线 → 人工终审 | 优惠券补发 |
| **MEDIUM** | 升级与兜底判断后暂停 → 人工确认 → 继续执行 → 人工终审 | 地址修改 |
| **HIGH** | 跳过自动执行和自动回复 → 直接升级人工处理 | 交易争议 |

### A2A-lite Agent Card

每个Agent通过 `agent_cards.json` 自描述能力、输入输出Schema、依赖关系和运行约束。编排器在启动时加载所有卡片，按拓扑排序决定执行顺序。

### SSE流式Trace

前端通过 `EventSource` 连接 `/api/tickets/{id}/ai-process-stream`，实时接收每个Agent的执行状态：
- `agent_start` → Agent开始工作
- `agent_thinking` → Agent进度消息  
- `agent_complete` → Agent完成（含结果+耗时）
- `workflow_paused` → 管线暂停等待人工确认
- `workflow_complete` → 管线完成（含完整结果）

## 八、常见问题

**Q: 启动后端报 `AuthenticationError: 401`？**

A: 未配置有效的 LLM API Key。请按第二章设置 `DEEPSEEK_API_KEY` 环境变量。支持 DeepSeek 和 Qwen（OpenAI 兼容接口）。

**Q: 前端页面空白？**

A: 确认后端已启动在 `localhost:8000`。当前前端 API 客户端默认访问 `http://localhost:8000/api`，所以端口或后端地址变更时需要同步修改 `frontend/src/api/index.ts`。

**Q: 旧版坐席工作台在哪里？**

A: 当前默认入口是模块 G+ 企业工单系统壳。旧版页面保留在 `http://localhost:5173/legacy/tickets`，详情页为 `/legacy/tickets/{id}`。

**Q: 可以直接用工单编号打开详情吗？**

A: 可以。`/tickets/:id` 同时支持内部 id 和可见工单编号，例如访问 `/tickets/T20260715001` 会解析到内部 id 并打开对应详情。

**Q: 工作池里看不到某个状态的工单？**

A: 左侧状态筛选和二级业务菜单会共同生效。先切回“待处理工单”或刷新 `/tickets` 首页；如果仍没有，检查后端 `GET /api/tickets` 返回的工单状态。

**Q: 为什么技术 Agent 名称不在主界面展示？**

A: 模块 G+ 已改为企业工单系统壳。主界面只展示坐席能理解的业务信息、证据链、处理日志和回单内容；底层 `IntakeAgent`、`ClassifierAgent` 等执行明细保留在“技术审计 / Agent 执行明细”折叠区。

**Q: Agent Copilot 会不会直接改业务状态？**

A: 不会。Copilot 只做启动处理、填入草稿、定位证据/缺失字段、打开审计和进入确认/复核等辅助动作。敏感确认必须走 `/confirm-action`，结案必须走主系统 `/close`。

**Q: 首页的“状态匹配率”是不是客户真实结案率？**

A: 不是。`closedLoopSuccessRate` 当前工程含义是“状态/预期结果匹配率”，用于评测 Agent 输出是否匹配标注预期，不代表真实客户闭环结案率。

**Q: 如何重置工单状态？**

A: 删除 `ai-engine/data/tickets.db`、`ai-engine/data/tickets.db-shm`、`ai-engine/data/tickets.db-wal` 后重启后端，系统会重新创建数据库并加载种子数据。删除前请确认你不需要保留本地处理记录。

**Q: 如何切换 LLM 模型？**

A: 设置 `LLM_MODEL` 环境变量，例如 PowerShell 中执行 `$env:LLM_MODEL="qwen-plus"`。

**Q: 如何添加新的工单场景？**

A: 1) 在 `data/tickets.json` 中添加工单数据；2) 在 `data/workflow_config.json` 中增加场景、必填字段、推荐工具和通知模板；3) 在 `agents/classifier_agent.py` 的 Prompt 中添加新的分类类型；4) 如需新工具，在 `data/tools.json` 中添加工具定义。

## 九、Demo 录制建议

汇报演示时建议按以下顺序：

1. **启动展示**（10秒）：展示两个终端窗口分别启动后端和前端。
2. **企业工单首页**（20秒）：展示左侧业务菜单树、顶部多标签、当前优先队列和评测摘要，说明这是企业工单系统中接入 Agent Copilot。
3. **场景1 - 优惠券补发**（60秒）：展示基本信息、客户与卡片信息、证据链、回单复核、复核并结案。
4. **场景2 - 地址修改**（45秒）：展示资料变更中风险、人工确认敏感操作、继续执行、回单复核。
5. **场景3 - 交易争议/盗刷疑似**（30秒）：展示高风险升级、人工接管原因、禁止自动结论。
6. **技术审计追问**（15秒）：展开“技术审计 / Agent 执行明细”，展示底层 Agent Trace 和耗时。
7. **架构亮点**（30秒）：Agent Card / 状态机 / SSE 流式事件 / Tool Registry / 证据链 / 评测指标。
