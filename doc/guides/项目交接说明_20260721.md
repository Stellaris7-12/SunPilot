# TicketAgent 项目交接说明（2026-07-21）

本文用于新会话、新开发者或后续任务接手时快速理解项目结构、业务逻辑、当前完成度和下一步开发重点。

## 1. 项目定位

TicketAgent 是一个信用卡工单多 Agent 处理系统，用来演示：

```text
客服电话/通话记录
  -> 发单生成标准工单
  -> 多 Agent 分析、补全、分诊、工具执行、风险兜底、回单生成
  -> SunPilot / Page Agent 辅助坐席填单、定位证据、准备复核
  -> 人工确认结案
```

当前项目不是生产银行系统直连版本，而是一个本地准生产业务沙盘。Mock Tools 表示外部业务系统能力适配层，用于演示工具选择、参数补全、证据编号、状态门禁和人工升级。

## 4. 后端业务逻辑

后端主链路由 `ai-engine/orchestrator/orchestrator.py` 控制，不是开放式多 Agent 自由协商，而是确定性业务流水线：

```text
加载工单
  -> ClassifierAgent：识别意图、场景、workflow、优先级
  -> IntakeAgent：抽取关键字段、识别缺失信息
  -> Field Enrichment：通过只读 Mock Tools 自动补全字段
  -> EscalationAgent：前置安全兜底，处理缺字段、高风险、未知场景
  -> ResolutionAgent：选择并调用业务工具
  -> EscalationAgent：后置复核工具失败、权限不足、冲突、高风险
  -> NotificationAgent：生成结构化回单和结案建议
  -> 持久化 AI Result / Trace / Tool Calls
```

五类业务 Agent：

| Agent | 主要职责 |
|---|---|
| `IntakeAgent` | 接单与信息提取，抽取字段、识别缺失信息 |
| `ClassifierAgent` | 工单分类、场景识别、workflow 和优先级判断 |
| `ResolutionAgent` | 选择业务工具、调用 Mock Tool、生成执行证据 |
| `EscalationAgent` | 缺字段、高风险、工具失败、人工确认和异常兜底 |
| `NotificationAgent` | 客户回单、内部通知、复核摘要、结案建议、回访预留 |

旧文件 `extract_agent.py`、`intent_agent.py`、`tool_agent.py`、`verify_agent.py`、`reply_agent.py` 目前是兼容 shim，短期保留，后续可在确认无依赖后清理。

## 5. 状态与结案边界

核心状态包括：

- `open`
- `processing`
- `pending_info`
- `pending_human_confirm`
- `pending_human_review`
- `escalated`
- `failed`
- `cancelled`
- `closed`

重要边界：

- 自动流程不直接把工单置为 `closed`。
- `NotificationAgent` 只输出 `closureSuggestion.canClose` 和最终回单建议。
- 实际结案必须通过 `POST /api/tickets/{ticket_id}/close`。
- 敏感资料变更、交易争议、投诉、征信异议等必须进入人工确认或升级路径。
- LLM 不允许覆盖确定性状态、证据编号、失败原因或结案规则。

## 6. 工具与 Mock 业务域

工具定义在：

- `ai-engine/data/tools.json`
- `ai-engine/tools/definitions.py`
- `ai-engine/tools/registry.py`
- `ai-engine/tools/mock_executor.py`

当前工具体系覆盖：

- 优惠券补发
- 客户资料变更
- 交易查询
- 权益查询
- 申请进度查询
- 以及 I3 后扩展的信息补全、业务核验、受控执行类工具

MockExecutor 会读取 Mock 业务域数据，例如客户、卡片、交易、权益、申请、权限和工具历史。若当前数据库缺少这些业务域数据，可能导致所有工单都找不到合适工具并升级人工。因此后续演示必须优先保证 MySQL 演示库是干净且完整的。

### 11.2 MySQL-only 口径

模块 K 后的数据库目标口径是：

```text
默认 MySQL 演示 + MySQL smoke，不再保留 SQLite 运行路径
```

当前约定：

- `ticket_agent` 用于本地开发/演示。
- `ticket_agent_test` 用于 I1/I2/I3 smoke，避免污染演示库。
- 本地连接可使用 MySQL `root`，密码来自系统环境变量 `MYSQL_ROOT_PASSWORD`。
- 旧 SQLite smoke 不再作为兼容目标；如需回归旧模块，应补写 MySQL-only smoke。

### 11.3 Mock Tools 大面积升级人工的常见原因

若所有工单都显示找不到合适 Mock Tools，优先检查：

- 当前后端是否连到正确的 MySQL 数据库
- MySQL `ticket_agent` 是否已执行 DDL
- 是否已运行演示数据重置脚本导入业务域数据
- 工单中的客户号、卡尾、交易流水、权益编码、申请单号是否能命中 Mock 业务域表
- `workflow_config.json` 中场景是否有 `recommended_tool`

不要一开始就假设 ResolutionAgent 选择逻辑坏了。

## 14. 答辩和对外交付口径

推荐表达：

- TicketAgent 的价值不是普通聊天，而是信用卡工单闭环：字段抽取、分类、工具执行、证据审计、人工门禁、结构化回单和可评测。
- MySQL 本地库是准生产演示沙盘，不是真实生产库。
- Mock Tools 是业务能力适配层的替身，用来证明工具调用、审计和状态门禁。
- `closedLoopSuccessRate` 是标注样本状态/预期结果匹配率，不是生产真实结案率。
- Page Agent / SunPilot 是受控页面执行层，不是通用网页机器人。
- 真实 ASR、Dispatcher、LangGraph、A2A、外部遗留系统自动化和生产权限体系均为后续路线。
